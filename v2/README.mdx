# Docs Core Principles

## Clear User Journeys
- Easy Navigation
- Clear Information Architecture
- Clean & Consistent Layouts & Information Structuring
- Comprehensive & Accurate Content Available
- Clear Content Structure - Presented in a logical user journey order (zero to hero)
- Clear Calls to Action
- Clear & Consistent Branding & Styling

## Future Proofing
Aiming to limit maintentance requirements & manual overhead on updates
- Snippets Wiki (see snippets folder)
- Automations (see separate section).

## Canonical Source of Truth
Given the decentralised nature of Livepeer, a lot of valuable information is siloed across the ecosystem.
These docs attempt to create a canonical source of truth for the Livepeer ecosystem and bring together information from multiple sources.
Through:
- Ecosystem HUBs where content aggregated
- Low-lift Contribution Pathways
- Embedding of key external documentation (eg. Orchestrator Spec, Developer Spec, etc.)
- Embedding of key Livepeer pages (eg. roadmap, protocol whitepaper )

## AI findable, searchable, integrateble
Increasingly, content is found by AI, not through search. These doc's attempt to both ensure accurate and up-to-date information is surfaced for AI queries as well as enable low-lift AI integrations.
- AI Assistant with additional processes [time-permitting]
    - ticket file [perhaps to a discord Q & A channel or to a github issue]
    - auto creation of PRs to contribute to doc's or other repos where errors are found
    -
- Comprehensive pages and guides in references to ensure all topics are covered
- AI Prompt pages for Quickstart Guides [future feature / WIP]

## Low-lift Contribution Pathways
- Automations to help smooth pathways to contributions (eg. Git, Markdown and Code understanding are often needed to contribute to docs - this is a barrier to contribution)
- Components Library to help provide consistent styling, branding and formatting across the docs
- Comprehensive and well-organised documentation codebase with READMEs like this one! (for AI & human contributors)

## Automations, Data Fetching & Scripts
Several automation pipelines are used in the docs to reduce manual overhead, ensure accuracy and provide the most up-to-date information possible.
They are run via GitHub Actions, n8n workflows and docs scripts.

They include:
- Fetching & embedding external documentation and specs from githubs (to ensure content remains updated) [WIP]
- Generating API documentation
- Fetching and displaying data (eg. github releases, forum top posts, blog posts, etc.)
- Fetching code information (eg. latest release version so code examples do not need a placeholder for example.)
- Feedback loops to and from discord (a highly siloed, inacceessible to AI and often disorganised source of valuable information)
