## Overview

The repo currently tracks roughly 200 scripts through the governance framework. Governed scripts are expected to carry a 9-field header, and the governance pipeline checks for drift at commit time, in PR CI, on post-merge governance sync, during weekly repair, and during on-demand runs, auto-repairing deterministic metadata where it can and surfacing the rest for a human.

## Trigger Chain

| Event | When | What runs | Auto-fixes? | Blocks? |
|---|---|---|---|---|
| Commit | Every `git commit` | `.githooks/pre-commit`, including `tests/unit/script-docs.test.js --staged --write --stage --autofill` | Placeholder header bootstrap for new scripts and script index refresh | Yes, when required tags are missing or still placeholders |
| PR | Every pull request to `main` or `docs-v2` | `.github/workflows/test-suite.yml` -> `npm --prefix tests run test:pr` -> `tests/run-pr-checks.js` | None | Yes, on failed changed-file checks |
| Push / merge to `docs-v2` | When governance-related paths land on `docs-v2` | `.github/workflows/governance-sync.yml` -> `tools/scripts/orchestrators/repair-governance.js` | Yes, safe governance repair plus follow-up PR creation when fixes are applied | No |
| Weekly repair | Monday 05:00 UTC | `.github/workflows/repair-governance.yml` -> `tools/scripts/orchestrators/repair-governance.js` in fix mode | Yes, safe governance repairs plus follow-up PR creation when files change | No |
| Weekly health check | Monday 06:00 UTC | `.github/workflows/content-health.yml` -> broader repo health and component drift audits | None | No |
| Manual GitHub dispatch | Any time from Actions | `repair-governance.yml` `workflow_dispatch` -> `repair-governance.js` in dry-run, fix, or report-only mode | Yes, in fix mode | No |
| Manual CLI | Any time locally | `cd tools && npm run audit:scripts` or `npm run repair:governance`, plus direct `node` modes for governance-specific audit and repair flags | Yes, in fix mode | N/A |

## Auto-Fixed vs Needs Human

**Auto-repaired (safe, deterministic):**

- Phantom rows removed from `tasks/reports/script-classifications.json`
- Missing classification rows added when a script already has valid governed metadata
- Existing classification rows updated when fields are already recoverable
- `@script` derived from the filename
- `@owner` defaulted to `docs`
- `@usage` derived from file kind and path
- `@scope` backfilled when it is already recoverable from governed metadata
- `@pipeline` corrected from actual trigger analysis
- `@category`, `@purpose`, `@needs`, and `@purpose-statement` backfilled only when classification data already exists
- Script indexes regenerated

**Needs human attention:**

- Choosing `@category` and `@purpose` for new scripts without classification data
- Choosing `@scope` when the right enum is not already recorded
- Writing `@needs`
- Writing `@purpose-statement`
- Deciding the intended execution surface, which determines pipeline wiring
- Adding or updating the matching row in `tasks/reports/script-classifications.json`

## Commands

Only the base repo audit and governance repair orchestration have npm aliases. The detailed governance audit and the extra repair modes use direct `node` invocations.

```bash
cd tools

npm run audit:scripts
npm run repair:governance

node scripts/validators/governance/audit-script-inventory.js --json --md
node scripts/validators/governance/audit-script-inventory.js --json --md --verbose
node scripts/validators/governance/audit-script-inventory.js --json --md --fix
node scripts/validators/governance/audit-script-inventory.js --json --md --fix --dry-run

node scripts/orchestrators/repair-governance.js --dry-run
node scripts/orchestrators/repair-governance.js --report-only
node scripts/orchestrators/repair-governance.js --auto-commit
```

## Adding a New Script

1. Create the script under `tools/scripts/`.
2. Add the full 9-field governance header up front: `@script`, `@category`, `@purpose`, `@scope`, `@owner`, `@needs`, `@purpose-statement`, `@pipeline`, and `@usage`.
3. Add or update the matching row in `tasks/reports/script-classifications.json` if the script should participate in governance repair.
4. Run `node tests/unit/script-docs.test.js --enforce-existing --write --rebuild-indexes`.
5. Commit your changes. Pre-commit will run `tests/unit/script-docs.test.js --staged --write --stage --autofill`. If you forgot a header completely, the hook can inject a placeholder bootstrap block, but you still need to replace it with real governance metadata before the commit will pass.
6. Open a PR. `tests/run-pr-checks.js` validates changed-file coverage in CI.
7. Merge to `docs-v2`. If governance-related paths changed, `governance-sync.yml` can open a follow-up repair PR. Weekly repair and manual repair remain the fallback for everything else.

Use the schema table below for valid enum choices. The current pre-commit hook tells you which tags are missing or placeholders, but it does not list the valid values for you.

## The 9-Field Header Schema

Treat all nine fields as required for a complete governed header. Some automation still distinguishes between minimum grading fields and full governance completeness, but new scripts should include the full set from the start.

| Field | Required? | Auto-repairable? | Valid values |
|---|---|---|---|
| `@script` | Yes | Yes | Filename-derived script identifier, usually the script basename |
| `@category` | Yes | Sometimes, from classification data | `validator`, `enforcer`, `remediator`, `generator`, `automation`, `utility`, `orchestrator` |
| `@purpose` | Yes | Sometimes, from classification data | `qa:content-quality`, `qa:link-integrity`, `qa:repo-health`, `governance:index-management`, `governance:agent-governance`, `governance:repo-health`, `feature:translation`, `feature:seo`, `infrastructure:data-feeds`, `infrastructure:pipeline-orchestration`, `tooling:api-spec`, `tooling:dev-tools` |
| `@scope` | Yes | Sometimes | `staged`, `changed`, `full-repo`, `v2-content`, `single-domain`, `single-file`, `external`, `generated-output` |
| `@owner` | Yes | Yes | `docs` |
| `@needs` | Yes | Sometimes, from classification data | Requirement IDs such as `E-C1`, `R-R29`, `F-R*`, or similar requirement references |
| `@purpose-statement` | Yes | Sometimes, from classification data | Short free-text explanation of what the script does |
| `@pipeline` | Yes | Yes | `P1`, `P2`, `P3`, `P5`, `P6`, `manual`, `indirect`, optionally with clarifying prose such as `P5 (scheduled, weekly)` |
| `@usage` | Yes | Yes | Executable command string such as `node ...`, `bash ...`, or `python3 ...` |
