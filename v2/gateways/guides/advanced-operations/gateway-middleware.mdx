The go-livepeer binary is a routing and payment engine. It does not authenticate your users, meter their usage, or route premium customers to different orchestrators. Those concerns belong to a middleware layer that you control, sitting between your customers and the gateway process.

This guide covers the common patterns for building that layer.

<Note>
This page covers the **request pipeline** — the layer between clients and your gateway. The **payment pipeline** (how your gateway settles with orchestrators using ETH or fiat credit) is covered in the [Payment Clearinghouse](/v2/gateways/guides/payments-and-pricing/clearinghouse-guide) guide.
</Note>

The middleware handles the customer-facing interface. The gateway handles the network interface. Each does one thing well.

| Concern | Middleware | go-livepeer |
|---------|-----------|-------------|
| API key validation | ✅ | ❌ |
| JWT / OAuth token verification | ✅ | ❌ |
| Per-customer rate limits | ✅ | ❌ |
| Usage tracking and billing | ✅ | ❌ |
| Customer-to-tier routing | ✅ | ❌ |
| Orchestrator selection | ❌ | ✅ |
| Probabilistic micropayments | ❌ | ✅ |
| On-chain settlement | ❌ | ✅ |

## Architecture patterns

Three deployment patterns cover most middleware use cases. Choose based on your existing infrastructure.

<Tabs>
  <Tab title="Reverse proxy (recommended)">
    The simplest and most maintainable pattern: an existing reverse proxy handles TLS, auth, and rate limiting before passing requests to the gateway.

    **Caddy with API key auth** (simplest):

    ```caddyfile
    ai-gateway.yourdomain.com {
        @authenticated {
            header Authorization "Bearer {$VALID_API_KEY}"
        }
        handle @authenticated {
            reverse_proxy localhost:8937
        }
        respond 401
    }
    ```

    **Nginx with external auth** (more flexible — delegates auth to your backend):

    ```nginx
    server {
        listen 443 ssl;
        server_name ai-gateway.yourdomain.com;

        location / {
            auth_request /auth;
            proxy_pass http://127.0.0.1:8937;
        }

        location = /auth {
            proxy_pass http://127.0.0.1:3000/verify-token;
            proxy_pass_request_body off;
            proxy_set_header Content-Length "";
            proxy_set_header X-Original-URI $request_uri;
        }
    }
    ```

    Your `/verify-token` endpoint returns `200` to allow or `401` to deny. It can also return headers that nginx forwards to the gateway, such as customer ID or tier.
  </Tab>
  <Tab title="API gateway">
    For multi-tenant deployments where you need per-customer rate limits, quota enforcement, and usage analytics, a dedicated API gateway layer fits well.

    **Kong** (open source, self-hosted):

    ```yaml
    # kong.yml — declarative config
    services:
      - name: livepeer-gateway
        url: http://livepeer-gateway:8937
        routes:
          - name: ai-inference
            paths:
              - /
        plugins:
          - name: key-auth
          - name: rate-limiting
            config:
              minute: 60
              policy: local
          - name: request-termination
            config:
              status_code: 429
              message: "Rate limit exceeded"
    ```

    **Traefik** (Docker-native, automatic service discovery):

    ```yaml
    # docker-compose.yml
    services:
      traefik:
        image: traefik:v3
        command:
          - "--providers.docker=true"
          - "--entrypoints.websecure.address=:443"

      livepeer-gateway:
        image: livepeer/go-livepeer:master
        labels:
          - "traefik.http.routers.livepeer.rule=Host(`ai-gateway.yourdomain.com`)"
          - "traefik.http.routers.livepeer.middlewares=auth@docker"
          - "traefik.http.middlewares.auth.basicauth.users=user:$$apr1$$..."
    ```
  </Tab>
  <Tab title="Custom proxy">
    For operators who need full control over routing logic, a lightweight custom proxy in Node.js, Go, or Python sits in front of the gateway.

    **Node.js example** (Express + http-proxy-middleware):

    ```javascript
    const express = require('express');
    const { createProxyMiddleware } = require('http-proxy-middleware');
    const app = express();

    // Auth middleware
    app.use(async (req, res, next) => {
      const apiKey = req.headers['x-api-key'];
      const customer = await validateApiKey(apiKey);
      if (!customer) return res.status(401).json({ error: 'Invalid API key' });

      // Attach customer context for routing and metering
      req.customer = customer;
      next();
    });

    // Usage metering
    app.use((req, res, next) => {
      const start = Date.now();
      res.on('finish', () => {
        recordUsage(req.customer.id, req.path, Date.now() - start);
      });
      next();
    });

    // Route to gateway
    app.use('/', createProxyMiddleware({
      target: 'http://localhost:8937',
      changeOrigin: true,
    }));

    app.listen(3000);
    ```
  </Tab>
</Tabs>

## Authentication patterns

<AccordionGroup>
  <Accordion title="API keys">
    The simplest approach. Issue each customer a unique API key, validate it on every request, and store it hashed in your database.

    ```
    Request: X-Api-Key: sk_live_abc123...
    Middleware: hash(sk_live_abc123...) → look up in DB → get customer record
    ```

    API keys are easy to rotate without requiring customers to re-authenticate. They are appropriate for server-to-server integrations where the key is stored securely on the client side.
  </Accordion>
  <Accordion title="JWT tokens">
    For consumer-facing applications where users authenticate via OAuth (Google, GitHub, etc.), issue short-lived JWT access tokens after authentication. Validate the JWT signature on every request without a database lookup.

    ```javascript
    const jwt = require('jsonwebtoken');

    function verifyToken(req, res, next) {
      const token = req.headers.authorization?.replace('Bearer ', '');
      try {
        const payload = jwt.verify(token, process.env.JWT_SECRET);
        req.customer = { id: payload.sub, tier: payload.tier };
        next();
      } catch {
        res.status(401).json({ error: 'Invalid token' });
      }
    }
    ```

    JWTs are appropriate for gateways that serve end-users directly through a web or mobile application.
  </Accordion>
  <Accordion title="Live AI webhook auth">
    For Live AI (video-to-video) workloads, go-livepeer supports webhook-based RTMP authentication. The gateway calls your webhook when an RTMP stream connects, and you return allow or deny.

    ```bash
    -liveAIAuthWebhookUrl https://your-service.example.com/rtmp-auth
    -liveAIAuthApiKey <internal-api-key-for-webhook>
    ```

    Your webhook receives the stream key and can validate it against your customer database before allowing the stream through.

## Rate limiting

Rate limiting protects your gateway from abuse and enforces fair usage across customers. Implement it in your middleware layer, not in go-livepeer.

The appropriate limits depend on your orchestrator pool capacity and customer agreements. A reasonable starting point for an AI inference gateway:

| Tier | Requests per minute | Concurrent sessions |
|------|--------------------|--------------------|
| Free | 10 | 1 |
| Standard | 60 | 5 |
| Premium | 300 | 20 |

Track limits in Redis for shared-memory rate limiting across multiple middleware instances:

```javascript
const rateLimit = require('express-rate-limit');
const RedisStore = require('rate-limit-redis');

const limiter = rateLimit({
  windowMs: 60 * 1000,
  max: (req) => req.customer.tier === 'premium' ? 300 : 60,
  store: new RedisStore({ client: redisClient }),
  keyGenerator: (req) => req.customer.id,
});
```

## Custom routing logic

The most powerful use of middleware is routing different customers or job types to different gateway instances, each configured with a distinct orchestrator pool.

```javascript
// Route premium customers to high-performance orchestrators
app.use('/process', async (req, res) => {
  const target = req.customer.tier === 'premium'
    ? 'http://localhost:8937'   // Gateway A: premium orch list, high price cap
    : 'http://localhost:8938';  // Gateway B: standard orch list, lower price cap

  proxyRequest(req, res, target);
});

// Route by pipeline type
app.use('/ai/*', (req, res) => proxyRequest(req, res, 'http://localhost:8937'));
app.use('/video/*', (req, res) => proxyRequest(req, res, 'http://localhost:8935'));
```

This pattern, combined with [tiered orchestrator selection](/v2/gateways/guides/advanced-operations/orchestrator-selection#tiering-strategy), allows you to deliver different SLA levels from one middleware deployment.

## Usage metering and billing

go-livepeer does not track how much each of your customers has consumed — it only tracks what you owe orchestrators. You need to meter customer usage in your middleware.

What to track per request:

- Customer ID
- Pipeline type (text-to-image, video transcoding, etc.)
- Input parameters (resolution, number of outputs, duration)
- Response time
- Success or failure

Store events in a time-series database or event queue, then aggregate for invoicing. For per-pixel billing (video transcoding), you can calculate usage from the resolution and frame count. For AI inference, count requests per model and pipeline.

```javascript
res.on('finish', () => {
  usageQueue.push({
    customerId: req.customer.id,
    pipeline: extractPipeline(req.path),
    durationMs: Date.now() - startTime,
    statusCode: res.statusCode,
    timestamp: new Date().toISOString(),
  });
});
```

## Middleware vs clearinghouse

A clearinghouse combines middleware concerns with payment pipeline concerns into a single managed service. The distinction:

| Concern | Your middleware | Clearinghouse |
|---------|----------------|--------------|
| Customer auth | ✅ you manage | ✅ managed for you |
| Usage billing | ✅ you manage | ✅ managed for you |
| ETH key custody | ❌ not your concern | ✅ managed for you |
| Orchestrator payments | ❌ go-livepeer handles | ✅ managed for you |
| Multi-tenant isolation | ✅ you build | ✅ built in |
| Fiat settlement | ❌ you build | ✅ built in |

If you want to avoid building the payment side, the [Payment Clearinghouse](/v2/gateways/guides/payments-and-pricing/clearinghouse-guide) guide covers how to delegate ETH custody and payment accounting to a clearinghouse service while keeping your middleware for auth and routing.
