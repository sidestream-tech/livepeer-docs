A gateway routes jobs to orchestrators rather than processing them locally, but it still has resource limits: session capacity, network throughput, and (for dual gateways) the GPU resources shared between video and AI workloads. This guide covers how to measure those limits, decide when you have reached them, and scale in the right direction.

<Note>
Session limits and GPU management apply primarily to orchestrators running compute locally. A pure routing gateway (off-chain AI, no local GPU) is limited mainly by network throughput and horizontal instance count. Jump to [Horizontal scaling](#horizontal-scaling) if that describes your setup.
</Note>

---

## Signs you need to scale

Before adjusting anything, confirm you actually have a capacity problem. These are the reliable indicators.

<AccordionGroup>
  <Accordion title="Session limit reached">
    The Prometheus metric `livepeer_current_sessions_total` approaching `livepeer_max_sessions_total` means the gateway is at capacity and will start rejecting new sessions. This is the clearest signal.

    Check it directly:

    ```bash
    curl http://localhost:9090/metrics | grep livepeer_current_sessions
    ```

    {/* REVIEW: Confirm exact Prometheus metric names `livepeer_current_sessions_total` and `livepeer_max_sessions_total` — Rick to verify */}
  </Accordion>
  <Accordion title="GPU memory pressure">
    For dual gateways, the `/hardware/stats` endpoint reports GPU utilisation and memory:

    ```bash
    curl http://localhost:8935/hardware/stats
    ```

    If VRAM usage is consistently above 85%, you are at risk of OOM failures on new model loads (AI) or performance degradation on concurrent transcoding segments (video).

    {/* REVIEW: Confirm `/hardware/stats` is exposed on gateway HTTP port in current go-livepeer — Rick to verify */}
  </Accordion>
  <Accordion title="Increasing latency under load">
    Rising transcoding latency or AI inference latency under load, combined with high orchestrator swap rates, suggests the orchestrators you are routing to are also under pressure. This is an orchestrator-side scaling problem, not a gateway problem. See [Orchestrator Selection](/v2/gateways/guides/advanced-operations/orchestrator-selection) for how to expand your orchestrator pool.
  </Accordion>
  <Accordion title="Client rejections">
    Errors with `OrchestratorCapped` in the gateway log indicate that a downstream orchestrator has hit its own session limit and rejected your job. Either expand your orchestrator pool or negotiate higher capacity with your preferred orchestrators.
  </Accordion>
</AccordionGroup>

## Session limits

Session limits define how many concurrent jobs the gateway (or its local orchestrator, for dual setups) will accept. The default is 10.

```bash
-maxSessions 20
```

Setting this too high causes performance degradation or OOM failures. Setting it too low wastes available capacity. The right value comes from benchmarking.

### Benchmarking with livepeer_bench

`livepeer_bench` tests transcoding throughput for a range of concurrent session counts. Run this on the machine that will perform local transcoding before setting `-maxSessions` in production.

```bash
# Download the standard transcoding options file
wget https://raw.githubusercontent.com/livepeer/go-livepeer/master/cmd/livepeer_bench/transcodingOptions.json

# Run the benchmark across 1 to 20 concurrent sessions
for i in {1..20}; do
    ./livepeer_bench \
        -in bbb/source.m3u8 \
        -transcodingOptions transcodingOptions.json \
        -nvidia 0 \
        -concurrentSessions $i |& grep "Duration Ratio" >> bench.log
done

cat bench.log
```

Read the output: find the last line where the real-time duration ratio is at or below 0.8. That concurrent session count is your hardware limit. Leave a 20% buffer for upload and download overhead.

```
Concurrent sessions 12: Duration Ratio 0.79  ← last line ≤ 0.8
Concurrent sessions 13: Duration Ratio 0.88
Concurrent sessions 14: Duration Ratio 0.97
```

In this example, set `-maxSessions 12`.

For machines with multiple identical GPUs, benchmark one GPU and multiply. For machines with different GPU models, benchmark each separately and use the lower value as the bottleneck.

<Tip>
Bandwidth can be a tighter limit than compute. Once you have a hardware session limit, estimate your transcoding bandwidth: each 1080p stream requires approximately 4–8 Mbps inbound and 12–24 Mbps outbound (for a 4-rung ABR ladder). If bandwidth is the bottleneck, it sets a lower ceiling than the hardware benchmark.
</Tip>

## GPU resource management

For dual gateways, video transcoding and AI inference compete for the same GPU. The hardware endpoints help you understand the current state.

```bash
# GPU model, VRAM, and system specs
curl http://localhost:8935/hardware/info

# Live GPU utilisation, memory, temperature
curl http://localhost:8935/hardware/stats
```

{/* REVIEW: Confirm `/hardware/info` and `/hardware/stats` endpoint paths are correct in current go-livepeer and whether they are exposed on the gateway HTTP port or a separate port — Rick to verify */}

VRAM allocation between video transcoding and AI inference is managed by the orchestrator, not the gateway. Video transcoding (NVENC/NVDEC) uses dedicated hardware encoder/decoder blocks on NVIDIA GPUs, which do not compete directly with CUDA cores used for AI inference. However, VRAM is shared, and large AI models can exhaust available memory.

As a rule of thumb:
- Reserve at least 8 GB VRAM per concurrent AI inference model
- Keep video transcoding sessions below your benchmarked hardware limit even when running AI jobs simultaneously
- Monitor temperature: sustained GPU temperature above 85°C indicates thermal throttling

## Dual gateway: split or share?

If you are running a dual gateway (video + AI on one node), the central decision is when to split those workloads onto separate nodes.

<Tabs>
  <Tab title="Decision guide">
    | Factor | Stay on single node | Split onto separate nodes |
    |--------|--------------------|-----------------------------|
    | Load | Low to medium, predictable | High, variable, or growing |
    | SLA requirements | Best-effort | Committed latency or uptime |
    | GPU VRAM | Model fits with transcoding headroom | Models exhaust VRAM under load |
    | Operational complexity | Prefer simplicity | Can manage two nodes |
    | Cost | Minimise hardware spend | Willing to pay for isolation |

    The practical trigger for splitting is usually one of two things: VRAM contention (AI models OOM when video load spikes), or SLA pressure (customers require guaranteed latency for one workload that the other workload interferes with).
  </Tab>
  <Tab title="Architecture after splitting">
    Once split, run two separate go-livepeer processes — one configured as a video gateway, one as an AI gateway — each with its own orchestrator list:

    ```bash
    # Video gateway (on-chain, port 8935, RTMP 1935)
    livepeer -gateway \
        -network arbitrum-one-mainnet \
        -rtmpAddr 0.0.0.0:1935 \
        -httpAddr 0.0.0.0:8935 \
        -orchAddr <video-orchestrators>

    # AI gateway (off-chain, port 8937)
    livepeer -gateway \
        -network offchain \
        -httpAddr 0.0.0.0:8937 \
        -orchAddr <ai-orchestrators> \
        -httpIngest
    ```

    Your middleware or load balancer routes incoming requests to the correct gateway based on job type.
  </Tab>
</Tabs>

**Off-chain AI gateways** scale horizontally without any special configuration. They are stateless: no ETH key, no session state. Add instances behind a standard HTTP load balancer and use the `/health` endpoint for health checks.

```yaml
# Example: three AI gateway instances behind nginx
upstream ai_gateways {
    server 10.0.0.1:8937;
    server 10.0.0.2:8937;
    server 10.0.0.3:8937;
}

server {
    location / {
        proxy_pass http://ai_gateways;
    }
}
```

**Video gateways** require sticky sessions for streaming continuity. A live RTMP stream cannot be moved between gateway instances mid-session. Configure your load balancer to use IP-hash or cookie-based session affinity for RTMP connections. HTTP API requests (non-streaming) can be round-robined normally.

**Dual gateways**: split into separate video and AI instances before scaling horizontally. Trying to horizontally scale a dual node adds complexity without solving the resource contention problem.

## Capacity planning

Estimate your required capacity before provisioning hardware or expanding your orchestrator pool.

<AccordionGroup>
  <Accordion title="Video transcoding capacity">
    Per GPU, the practical limit for 1080p → 720p/480p/360p transcoding is approximately 8–12 concurrent sessions on a modern NVIDIA T4, and 15–25 on an RTX 3080 or equivalent. Run `livepeer_bench` on your specific hardware to get a precise number.

    Multiply by the number of GPUs in your orchestrator pool to get total network capacity. Your gateway can route to all of them.

    For deposit sizing: video transcoding payments are per-pixel. Estimate your expected pixel throughput (resolution × frame rate × concurrent sessions × hours per day) and size your Arbitrum deposit to cover at least 24 hours of expected traffic.
  </Accordion>
  <Accordion title="AI inference capacity">
    AI inference capacity depends entirely on the model and the orchestrators in your pool. FLUX.1-dev requires approximately 12–16 GB VRAM per concurrent inference; smaller SD 1.5 models can fit in 6 GB.

    For off-chain AI gateways, your bottleneck is the number of AI-capable orchestrators you have access to. Expand your `-orchAddr` list or use a discovery URL that returns a larger pool.

    Set a monitoring alert at 70% of observed capacity. This gives you time to provision additional orchestrators before you hit the ceiling.
  </Accordion>
  <Accordion title="Alert thresholds">
    | Metric | Warning | Critical |
    |--------|---------|----------|
    | Session utilisation | 70% of `-maxSessions` | 90% of `-maxSessions` |
    | GPU memory | 75% VRAM used | 90% VRAM used |
    | GPU temperature | 80°C | 87°C |
    | Orchestrator swap rate | 2× baseline | 5× baseline |
    | ETH deposit balance | 20% remaining | 5% remaining |
  </Accordion>
</AccordionGroup>
