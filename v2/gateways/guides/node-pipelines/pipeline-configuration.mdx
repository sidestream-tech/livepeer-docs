This page consolidates the configuration reference for all three pipeline types. It covers the encoding ladder JSON for video transcoding, AI routing flags, per-pipeline pricing, and the resource allocation decisions that matter when running a dual gateway. This is post-setup tuning — for initial installation, see the Setup section for your gateway type.

<Note>
  This page covers configuration that changes how pipelines behave once your gateway is running. For flag definitions and full CLI reference, see Resources → Configuration Flags.
</Note>

    1. VIDEO: TRANSCODING PROFILES
    ============================================================ */}

## Video transcoding profiles

The transcoding profile defines which output renditions your gateway produces for every incoming stream. You pass a profile to the gateway via the `-transcodingOptions` flag, either as a comma-separated list of built-in presets or as a path to a JSON file.

### Built-in presets

The simplest option. Presets follow the pattern `P<height>p<fps>fps<aspect>`:

```bash
-transcodingOptions P240p30fps16x9,P360p30fps16x9,P720p30fps16x9
```

Use built-in presets for development or when the defaults match your use case. For production, a JSON profile gives you control over bitrate, codec, and GOP size.

### JSON profile format

Create a `transcodingOptions.json` file and reference it with `-transcodingOptions /path/to/transcodingOptions.json`.

```json
[
  {
    "name": "1080p",
    "width": 1920,
    "height": 1080,
    "bitrate": 6500000,
    "fps": 0,
    "profile": "h264constrainedhigh",
    "gop": "1"
  },
  {
    "name": "720p",
    "width": 1280,
    "height": 720,
    "bitrate": 3000000,
    "fps": 0,
    "profile": "h264constrainedhigh",
    "gop": "1"
  },
  {
    "name": "480p",
    "width": 854,
    "height": 480,
    "bitrate": 1600000,
    "fps": 0,
    "profile": "h264constrainedhigh",
    "gop": "1"
  },
  {
    "name": "240p",
    "width": 426,
    "height": 240,
    "bitrate": 600000,
    "fps": 0,
    "profile": "h264constrainedhigh",
    "gop": "1"
  }
]
```

**Field reference:**

| Field | Required | Notes |
|---|---|---|
| `width` | Yes | Output width in pixels |
| `height` | Yes | Output height in pixels |
| `bitrate` | Yes | Output bitrate in bits per second |
| `name` | No | Human-readable label, used in HLS manifest |
| `fps` | No | Target frames per second; `0` preserves source FPS |
| `profile` | No | H.264 profile: `h264constrainedhigh`, `h264high`, `h264main`, `h264baseline` |
| `gop` | No | GOP size in seconds or `"intra"` for keyframe-only; lower values reduce latency |

### Per-platform profile setup

<Tabs>
  <Tab title="Docker">
    Mount the JSON file as a volume and reference it via a path inside the container:

    ```yaml
    # docker-compose.yml
    services:
      gateway:
        image: livepeer/go-livepeer:master
        volumes:
          - ./transcodingOptions.json:/config/transcodingOptions.json
          - gateway-data:/root/.lpData
        command: |
          -gateway
          -transcodingOptions /config/transcodingOptions.json
          -rtmpAddr 0.0.0.0:1935
          -httpAddr 0.0.0.0:8935
          -orchAddr https://orch1.example.com:8935
    ```

    Alternatively, write directly to the gateway's data volume:

    ```bash
    nano /var/lib/docker/volumes/gateway-lpData/_data/transcodingOptions.json
    ```

    And reference it at the container path:

    ```
    -transcodingOptions /root/.lpData/transcodingOptions.json
    ```
  </Tab>
  <Tab title="Linux (systemd)">
    Create the profile file in your chosen config directory:

    ```bash
    sudo nano /usr/local/bin/lptConfig/transcodingOptions.json
    ```

    Reference it in your systemd service file (`/etc/systemd/system/livepeer.service`):

    ```ini
    ExecStart=/usr/local/bin/livepeer \
      -gateway \
      -transcodingOptions /usr/local/bin/lptConfig/transcodingOptions.json \
      -rtmpAddr 0.0.0.0:1935 \
      -httpAddr 0.0.0.0:8935
    ```

    Reload systemd after any service file change:

    ```bash
    sudo systemctl daemon-reload && sudo systemctl restart livepeer
    ```
  </Tab>
  <Tab title="Windows">
    Create the profile file in the `.lpData` directory:

    ```
    C:\Users\%USERNAME%\.lpData\transcodingOptions.json
    ```

    Reference it in your startup `.bat` file:

    ```bat
    livepeer.exe ^
      -gateway ^
      -transcodingOptions C:\Users\%USERNAME%\.lpData\transcodingOptions.json ^
      -rtmpAddr 0.0.0.0:1935 ^
      -httpAddr 0.0.0.0:8935
    ```
  </Tab>
</Tabs>

### Common profile recipes

<AccordionGroup>
  <Accordion title="Standard adaptive bitrate ladder (recommended for most deployments)" icon="layer-group">
    Four renditions covering mobile to desktop. Viewers' players switch automatically based on available bandwidth.

    ```json
    [
      { "name": "720p",  "width": 1280, "height": 720,  "bitrate": 3000000, "fps": 0, "profile": "h264constrainedhigh", "gop": "1" },
      { "name": "480p",  "width": 854,  "height": 480,  "bitrate": 1600000, "fps": 0, "profile": "h264constrainedhigh", "gop": "1" },
      { "name": "360p",  "width": 640,  "height": 360,  "bitrate": 800000,  "fps": 0, "profile": "h264constrainedhigh", "gop": "1" },
      { "name": "240p",  "width": 426,  "height": 240,  "bitrate": 400000,  "fps": 0, "profile": "h264constrainedhigh", "gop": "1" }
    ]
    ```
  </Accordion>

  <Accordion title="Low-latency single rendition" icon="bolt">
    For use cases where you need a single output with minimal end-to-end latency (for example, feeding a monitoring pipeline or an AI processing step).

    ```json
    [
      { "name": "480p-lowlat", "width": 854, "height": 480, "bitrate": 1200000, "fps": 0, "profile": "h264baseline", "gop": "0.5" }
    ]
    ```

    `h264baseline` is more compatible with edge players. `"gop": "0.5"` halves the default GOP, reducing segment duration and latency.
  </Accordion>

  <Accordion title="High-quality archive (1080p)" icon="archive">
    Single high-quality rendition for recording or post-production.

    ```json
    [
      { "name": "1080p-hq", "width": 1920, "height": 1080, "bitrate": 8000000, "fps": 0, "profile": "h264high", "gop": "2" }
    ]
    ```

    Higher bitrate and larger GOP (less overhead, better compression). Not suitable for live playback — use with `-objectStore` for archival.
  </Accordion>
</AccordionGroup>

## Video pricing flags

| Flag | Default | Description |
|---|---|---|
| `-maxPricePerUnit` | `0` | Maximum wei per pixel your gateway will pay orchestrators. `0` accepts any price. |
| `-pixelsPerUnit` | `1` | How many pixels constitute one "unit" for pricing purposes. |
| `-ignoreMaxPriceIfNeeded` | `false` | If `true`, the gateway accepts any price when no orchestrators are available under the ceiling. |

**Setting a price ceiling for production:**

```bash
-maxPricePerUnit 1000
```

A value of `1000` wei per pixel is a reasonable starting point for production. Adjust based on orchestrator availability — if jobs are failing because no orchestrators are below your ceiling, raise it. Use `livepeer_cli → Option 9` to see currently advertised prices.

**USD-denominated pricing (Chainlink oracle):**

```bash
-priceFeedAddr <CHAINLINK_ETH_USD_ADDRESS>
```

{/* REVIEW: confirm Chainlink contract address on Arbitrum One for ETH/USD feed — verify with Rick/Mehrdad before publishing */}

When set, the gateway interprets `-maxPricePerUnit` as USD cents per pixel (converted to wei at current ETH price) rather than absolute wei. This stabilises your cost ceiling against ETH price volatility.

## AI routing flags

<AccordionGroup>
  <Accordion title="-orchAddr (required)" icon="server">
    Comma-separated list of AI-capable orchestrator addresses. This is the only discovery mechanism for off-chain AI gateways.

    ```bash
    -orchAddr https://orch1.example.com:8935,https://orch2.example.com:8935
    ```

    Format: `scheme://host:port`. The port is required (default orchestrator port is 8935 for the gateway-facing API). Add more orchestrators for redundancy and load distribution.
  </Accordion>

  <Accordion title="-httpIngest (required)" icon="arrow-down-to-bracket">
    Enables the HTTP endpoint that accepts AI inference requests. Without this flag, the AI pipeline API at port 8937 will not accept incoming requests.

    ```bash
    -httpIngest
    ```

    This flag has no value — it is a boolean toggle.
  </Accordion>

  <Accordion title="-httpAddr (AI port)" icon="network-wired">
    Sets the address and port the AI pipeline API listens on.

    ```bash
    -httpAddr 0.0.0.0:8937
    ```

    Use `0.0.0.0` to accept connections from all interfaces. Use `127.0.0.1` to restrict to localhost only.

    <Warning>
      The AI gateway default port is `8937`, not `8935`. If you are running a dual gateway, both `8935` (video) and `8937` (AI) need to be exposed.
    </Warning>
  </Accordion>

  <Accordion title="-aiProcessingRetryTimeout" icon="clock-rotate-left">
    Controls how long the gateway waits before retrying a failed AI request with another orchestrator. Accepts Go duration format.

    ```bash
    -aiProcessingRetryTimeout 30s
    ```

    Increase this for pipelines that use slow-loading (cold) models, or for BYOC containers with longer boot times. Setting it too low causes the gateway to retry before the orchestrator has had time to load the model, resulting in multiple failed attempts.

    Recommended values by workload:
    - Fast warm models (Lightning, LCM): `15s`
    - Standard diffusion models: `30s`
    - Cold BYOC containers or large models: `60s` or higher
  </Accordion>
</AccordionGroup>

### Complete AI gateway command

```bash
livepeer \
  -datadir ~/.lpData2 \
  -gateway \
  -orchAddr https://orch1.example.com:8935,https://orch2.example.com:8935 \
  -httpAddr 0.0.0.0:8937 \
  -httpIngest \
  -aiProcessingRetryTimeout 30s \
  -maxPricePerUnit 0 \
  -v 6
```

## AI model pricing: the orchestrator's aiModels.json

While you do not edit `aiModels.json` as a gateway operator, understanding its structure helps you interpret the pricing model and know what prices orchestrators are advertising.

Orchestrators configure per-model pricing in `~/.lpData/aiModels.json`:

```json
[
  {
    "pipeline": "text-to-image",
    "model_id": "SG161222/RealVisXL_V4.0_Lightning",
    "price_per_unit": 4768371,
    "warm": true,
    "optimization_flags": {
      "SFAST": true
    }
  },
  {
    "pipeline": "audio-to-text",
    "model_id": "openai/whisper-large-v3",
    "price_per_unit": 12882811,
    "pixels_per_unit": 1
  },
  {
    "pipeline": "upscale",
    "model_id": "stabilityai/stable-diffusion-x4-upscaler",
    "price_per_unit": "0.5e-2USD",
    "warm": true
  }
]
```

**Key pricing fields:**

| Field | Notes |
|---|---|
| `pipeline` | The pipeline type this model serves |
| `model_id` | Hugging Face model ID — must match what you send in requests |
| `price_per_unit` | Price in wei per pixel (integers), or a USD string like `"0.5e-2USD"` using Chainlink oracle |
| `warm` | If `true`, the model is preloaded on GPU — expect fast responses |
| `optimization_flags.SFAST` | Enables Stable Fast optimisation — up to 25% speed increase on `image-to-video` |
| `optimization_flags.DEEPCACHE` | Enables DeepCache — up to 50% speed increase on `text-to-image` with standard diffusion models. **Do not use with Lightning or Turbo models.** |

As a gateway operator, your `-maxPricePerUnit` or `-maxPricePerCapability` flags set the ceiling against these orchestrator-advertised prices.

## Per-capability AI pricing

For AI pipelines, you can set per-pipeline-and-model price caps via `-maxPricePerCapability`. This accepts a JSON configuration.

```bash
-maxPricePerCapability /path/to/aiPricing.json
```

Example `aiPricing.json`:

```json
{
  "text-to-image": {
    "SG161222/RealVisXL_V4.0_Lightning": 5000000,
    "stabilityai/stable-diffusion-xl-base-1.0": 8000000
  },
  "audio-to-text": {
    "openai/whisper-large-v3": 15000000
  }
}
```

This lets you set different price ceilings per pipeline and model, rather than a single ceiling for all AI work. Useful when running a mixed workload where some pipelines are more compute-intensive than others.

## Dual gateway: resource allocation

A dual gateway runs both video transcoding and AI inference from the same node. The main risk is resource contention — video and AI workloads competing for GPU memory, CPU, or network bandwidth.

<Tabs>
  <Tab title="Port configuration">
    Both ports must be configured and exposed:

    ```bash
    -rtmpAddr 0.0.0.0:1935  # RTMP ingest (video)
    -httpAddr 0.0.0.0:8935  # Video API and HLS
    -httpAddr 0.0.0.0:8937  # AI pipeline API
    ```

    {/* REVIEW: confirm whether both -httpAddr flags can coexist or if dual mode requires a different flag — verify with Rick */}

    In Docker, both ports need to be mapped:

    ```yaml
    ports:
      - "1935:1935"  # RTMP
      - "8935:8935"  # Video API
      - "8937:8937"  # AI API
    ```
  </Tab>
  <Tab title="Resource contention signals">
    Use the `/hardware/stats` endpoint to monitor GPU state when running both workloads:

    ```bash
    curl http://localhost:8935/hardware/stats
    ```

    Signs that contention is becoming a problem:
    - GPU memory utilisation consistently above 85%
    - Increasing `livepeer_orchestrator_swaps` rate on video jobs
    - Rising AI request latency or timeout errors
    - `livepeer_discovery_errors_total` climbing for AI

    If you see these signals, separate video and AI onto different nodes.
  </Tab>
  <Tab title="When to split">
    Run a dual gateway when:
    - Your total GPU memory comfortably fits both a warm AI model and the transcoding pipeline
    - Combined throughput is well below session limits
    - You are experimenting or building toward separate nodes

    Split to separate nodes when:
    - Either workload is latency-sensitive and the other is affecting it
    - GPU memory pressure is causing model eviction
    - You want independent scaling of video and AI capacity
  </Tab>
</Tabs>

## Quality and cost tradeoffs

<AccordionGroup>
  <Accordion title="Video: resolution vs cost" icon="film">
    Every output rendition adds pixels. Adding a 1080p rendition to a 720p+480p+360p ladder roughly doubles the pixel count (and cost). Evaluate whether viewers are actually watching at 1080p before adding it to your ladder. For mobile-first applications, a 480p+360p+240p ladder may be sufficient and significantly cheaper.

    Estimated per-segment cost scales linearly with total output pixels, not with the number of renditions.
  </Accordion>
  <Accordion title="Video: price ceiling vs orchestrator availability" icon="scale-balanced">
    A lower `-maxPricePerUnit` reduces cost but shrinks the set of available orchestrators. If your ceiling is below the market rate, jobs will fail silently. Use `-ignoreMaxPriceIfNeeded` as a safety valve — it accepts any price when no orchestrators are available under your ceiling, preventing job failure at the cost of occasionally overpaying.

    Check `livepeer_cli → Option 9` or the Livepeer Explorer for current orchestrator price distributions before setting your ceiling.
  </Accordion>
  <Accordion title="AI: inference steps vs latency" icon="brain">
    For diffusion models, inference steps directly control output quality and inference time. Lightning and LCM models are designed for 4–8 steps. Standard SDXL models need 20–50. Choosing the wrong model family with incorrect step counts produces either very slow responses (too many steps) or degraded image quality (too few steps on a standard model).
  </Accordion>
  <Accordion title="AI: warm vs cold models" icon="thermometer-half">
    Requesting a model that is not warm on any connected orchestrator will result in a slow first response (the model must load from disk or download). For latency-sensitive applications, check `tools.livepeer.cloud/ai/network-capabilities` to confirm your chosen model is warm on at least one connected orchestrator before making it the default.
  </Accordion>
</AccordionGroup>

## Verifying your configuration

**Verify video transcoding profiles are applied:**

Check the startup log for a line confirming your profile is loaded:

```
I0501 11:07:47.609839  1 mediaserver.go:201] Transcode Job Type: [{480p0 ...} {720p0 ...}]
```

Send a test stream and check the HLS manifest for the expected renditions:

```bash
ffprobe http://localhost:8935/<stream-key>.m3u8
```

**Verify AI pipeline is accepting requests:**

```bash
curl -s http://localhost:8937/health
```

A healthy response confirms the AI API is running. Then send a test inference request:

```bash
curl -X POST http://localhost:8937/text-to-image \
  -H "Content-Type: application/json" \
  -d '{
    "model_id": "SG161222/RealVisXL_V4.0_Lightning",
    "prompt": "a test image",
    "width": 512,
    "height": 512,
    "num_inference_steps": 4,
    "guidance_scale": 1.5
  }'
```

A JSON response with an `images` array confirms routing is working.

**Check effective configuration via livepeer_cli:**

```bash
livepeer_cli -host 127.0.0.1 -http 5935
```

Option 1 ("Get node status") shows your current deposit, reserve, connected orchestrators, and active sessions.
