Run these checks the moment you start your gateway, after any configuration change, or when something seems wrong. Health checks are quick — full observability setup is in [Monitoring Setup](/v2/gateways/guides/monitoring-troubleshooting/monitoring-setup).

<Info>
  Each gateway type has different health requirements. Jump to your type: [Video](#video-gateway-on-chain) · [AI](#ai-gateway-off-chain) · [Dual](#dual-gateway)
</Info>

## Universal Checks — All Gateway Types

These two endpoints work on every gateway type. Start here.

### HTTP Health Endpoint

```bash
# Video gateway (default port 8935)
curl http://localhost:8935/health

# AI gateway (default port 8937)
curl http://localhost:8937/health
```

A healthy gateway returns HTTP `200` with a JSON body. Any non-200 response or connection refused means the process is not running or the port is wrong.

```json
{
  "status": "OK"
}
```

### Node Status via livepeer_cli

```bash
# Standard (binary on host)
livepeer_cli -host 127.0.0.1 -http 5935

# Docker container
docker exec -it <container_name> livepeer_cli -host <container_hostname> -http 5935
```

Select **"Get node status"** from the menu. The status output shows your node version, network, ETH address, current sessions, and (for on-chain nodes) deposit and reserve balances.

### Hardware Status

```bash
# GPU utilisation, memory, temperature
curl http://localhost:<PORT>/hardware/stats

# GPU model and system info
curl http://localhost:<PORT>/hardware/info
```

## Video Gateway (on-chain)

A video gateway is healthy when all four of the following are true:

1. `/health` returns 200
2. ETH deposit is non-zero
3. At least one orchestrator is reachable
4. Arbitrum RPC is responding

### Check ETH Deposit and Reserve

```bash
livepeer_cli -host 127.0.0.1 -http 5935
# Select: "Get node status"
```

Look for the **BROADCASTER STATS** section in the output. You need:
- `Deposit` — non-zero (gateway cannot issue tickets without this)
- `Reserve` — non-zero (required for ticket redemption float)

<Warning>
  When your deposit reaches zero, the gateway stops routing jobs — no error is shown to the stream sender. Monitor your balance proactively. Community-recommended minimum for testing: **0.1 ETH total** (approximately 0.07 ETH deposit + 0.03 ETH reserve).

  {/* REVIEW: Rick / Mehrdad — confirm current recommended minimum deposit/reserve amounts */}
</Warning>

If deposit shows zero after bridging ETH, see [Fund Your Gateway](/v2/gateways/guides/payments-pricing/fund-your-gateway) — bridging does not automatically allocate funds.

### Test Arbitrum RPC Connectivity

```bash
curl -X POST <YOUR_ETH_URL> \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"eth_blockNumber","params":[],"id":1}'
```

A working RPC returns a JSON response with a hex block number. If the request times out or returns an error, your RPC is rate-limited or unavailable. See [Troubleshooting → Arbitrum RPC failures](/v2/gateways/guides/monitoring-troubleshooting/troubleshooting#arbitrum-rpc-connection-failing).

### Test RTMP Ingest (Optional)

```bash
# Push a test stream to the RTMP port (default 1935)
ffmpeg -re -f lavfi -i testsrc=size=1280x720:rate=30 \
  -c:v libx264 -preset ultrafast \
  -f flv rtmp://localhost:1935/live/test
```

If the gateway is healthy, you will see transcoding activity in logs (`-v 6` for verbose). If the stream immediately drops, check deposit balance and orchestrator connectivity first.

## AI Gateway (off-chain)

An AI gateway is healthy when:

1. `/health` on port 8937 returns 200
2. At least one orchestrator in `-orchAddr` is reachable
3. A test job succeeds end-to-end

### Check Orchestrator Reachability

```bash
# Verify each orchestrator in your -orchAddr list is responding
curl https://<orchestrator-address>:<port>/getOrchestratorInfo
```

{/* REVIEW: Rick / j0sh — confirm /getOrchestratorInfo is the correct endpoint for capability queries */}

If an orchestrator is unreachable, jobs silently fail to route. Add or replace addresses in `-orchAddr` with known-active AI orchestrators. Ask in [Discord #local-gateways](https://discord.gg/livepeer) for current AI-capable orchestrator addresses.

### Run a Test Job

```bash
curl -X POST http://localhost:8937/text-to-image \
  -H "Content-Type: application/json" \
  -d '{
    "model_id": "ByteDance/SDXL-Lightning",
    "prompt": "a simple geometric test pattern",
    "width": 512,
    "height": 512,
    "num_images_per_prompt": 1
  }'
```

{/* REVIEW: Peter (AI SPE) — confirm ByteDance/SDXL-Lightning is still a reliable model for onboarding tests */}

A successful response returns a base64-encoded image. An error typically means either the orchestrator does not have this model loaded or your `-orchAddr` is misconfigured.

<Note>
  The AI gateway does not require ETH, LPT, or an Arbitrum connection. If you are seeing RPC-related errors on an AI-only gateway, you may have accidentally included `-network arbitrum-one-mainnet` in your startup flags.
</Note>

## Dual Gateway

A dual gateway runs both video and AI workloads simultaneously. Run all video checks and all AI checks above, then add:

### Check Both Ports

```bash
# Video port
curl http://localhost:8935/health

# AI port
curl http://localhost:8937/health
```

Both must return 200.

### Check Hardware Under Load

```bash
curl http://localhost:8935/hardware/stats
```

Watch for:
- `gpu_memory_used` approaching `gpu_memory_total` — model loading will fail when GPU VRAM is full
- High GPU utilisation when no jobs are running — indicates a stuck process

When both video and AI workloads are active simultaneously, memory pressure is the most common cause of degraded performance. If you see consistent GPU memory exhaustion, consider splitting video and AI onto separate nodes.

## Automate Health Checks

### Simple Cron Script

```bash
#!/usr/bin/env bash
# gateway-health.sh — run every 5 minutes via cron

GATEWAY_PORT="${1:-8935}"
ALERT_EMAIL="you@example.com"

if ! curl --silent --fail --max-time 5 http://localhost:${GATEWAY_PORT}/health > /dev/null 2>&1; then
  echo "Gateway on port ${GATEWAY_PORT} is not responding" | \
    mail -s "ALERT: Livepeer Gateway Down" "$ALERT_EMAIL"
fi
```

```bash
# Add to crontab
*/5 * * * * /path/to/gateway-health.sh 8935
```

### Docker HEALTHCHECK

```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
  CMD curl --silent --fail http://localhost:8935/health || exit 1
```

For full observability — time-series metrics, alerting, and dashboards — see [Monitoring Setup](/v2/gateways/guides/monitoring-troubleshooting/monitoring-setup).
