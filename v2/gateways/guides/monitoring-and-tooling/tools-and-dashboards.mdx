Four tools form the gateway operator's monitoring toolkit. Each answers different questions — start with the one that matches what you are trying to find.

## Tool Overview

| Tool | What it shows | Gateway types | Access |
|------|--------------|---------------|--------|
| **Livepeer Explorer** | On-chain data: orchestrators, staking, gateway registrations | Video, Dual | [explorer.livepeer.org](https://explorer.livepeer.org) |
| **Livepeer Tools** | Gateway performance metrics, network-wide AI view | All types | [tools.livepeer.cloud](https://tools.livepeer.cloud) |
| **Arbiscan** | Contract events: ticket redemptions, deposit changes, gas | Video, Dual | [arbiscan.io](https://arbiscan.io) |
| **livepeer_cli** | Local node status: deposit, reserve, sessions, config | All types | CLI on host or Docker |
| **CLI HTTP API** | Programmatic node status and configuration | All types | `http://localhost:5935/*` |

## Livepeer Explorer

The Livepeer Explorer is the primary tool for on-chain data about the network. It surfaces orchestrator information, staking, and gateway registrations from Arbitrum One.

<Tip>
  Gateway-specific views in the Explorer are currently in development. Some gateway metrics may not yet be visible. Check back as this feature is released.
</Tip>

### What Gateway Operators Use Explorer For

**Orchestrator discovery:**
Browse active orchestrators, their capacity, advertised price per pixel, stake, and uptime. Use this to evaluate which orchestrators to add to your `-orchAddr` list or remove when performance degrades.

**Verify your gateway is registered:**
If you are running an on-chain gateway, your ETH address should appear in the Explorer after your first successful deposit transaction. Search for your gateway address to see on-chain activity.

**Cross-reference with Arbiscan:**
Explorer aggregates on-chain data in a readable format. For raw transaction and event data (ticket redemptions, exact wei amounts), Arbiscan gives more granular detail.

## Livepeer Tools Dashboard

Livepeer Tools is a dashboard built by the community SPE (Service Provider Entity) ecosystem. It provides a network-wide view of gateway and orchestrator performance metrics.

{/* REVIEW: DeFine / Titan-Node — confirm what data Livepeer Tools currently exposes for gateway operators, and whether it covers AI metrics specifically */}

### What Gateway Operators Use Livepeer Tools For

**Network-wide AI orchestrator view:**
See which AI orchestrators are active, what pipelines they support, and approximate load. Useful for choosing orchestrators to add to your AI gateway's `-orchAddr` list.

**Gateway performance overview:**
Compare your gateway's performance against network averages. Identify whether problems (high latency, low success rates) are localised to your setup or network-wide.

**Historical metrics:**
Unlike livepeer_cli which shows current state only, Livepeer Tools provides time-series charts for trend analysis.

## Arbiscan — On-Chain Event Monitoring

Arbiscan is the block explorer for Arbitrum One. It is the authoritative source for what has actually happened on-chain — contract event logs, transaction history, and token transfers.

**Applies to:** Video gateways and dual gateways only. Off-chain AI gateways have no on-chain activity.

### Key Use Cases

**Monitor deposit and reserve changes:**
Search your gateway address on Arbiscan to see every `FundDeposit` and `FundReserve` event — confirming your livepeer_cli deposits landed on-chain.

**Track ticket redemptions:**
Every time an orchestrator redeems a winning ticket against your deposit, a `RedeemWinningTicket` event fires. Arbiscan lets you see: which orchestrator redeemed, how much ETH they received, and at what gas cost.

**Detect unexpected drain:**
If your deposit is depleting faster than expected, Arbiscan transaction history shows the redemption events with timestamps — letting you identify when and by which orchestrators your deposit was drawn down.

**Set up address watchlist:**
Create an Arbiscan account and add your gateway address to your watchlist for email notifications on each transaction.

→ For detailed on-chain debugging workflows, see [On-Chain Debugging](/v2/gateways/guides/monitoring-and-tooling/on-chain-debugging).

→ For TicketBroker and all Livepeer contract addresses, see [Contract Addresses](/v2/gateways/resources/technical/contract-addresses).

## livepeer_cli

`livepeer_cli` is the local command-line interface for your running gateway node. It communicates with the node over HTTP on port 5935 (the CLI port, not the gateway port).

### Starting livepeer_cli

```bash
# Binary installed on host
livepeer_cli -host 127.0.0.1 -http 5935

# Docker container
docker exec -it <container_name> livepeer_cli -host <container_hostname> -http 5935
```

{/* REVIEW: Rick — confirm CLI port default is 5935 in current builds and that -host flag is still required */}

### Key Commands for Gateway Operators

**Get node status** — the first thing to check:

```
Menu option: "Get node status"
```

Output includes:
- Node version and network
- ETH account address
- Current session count
- (On-chain only) Deposit and reserve balance in ETH
- (On-chain only) Connected orchestrators

**Set broadcast config** — update pricing at runtime:

```
Menu option: "Set broadcast config"
```

Lets you adjust `-maxPricePerUnit` and transcoding profiles without restarting the node. Changes take effect on the next job.

**Deposit broadcasting funds** (Option 11):

```
Menu option: "Deposit broadcasting funds" (livepeer_cli option 11)
```

Triggers a deposit and reserve transaction to the TicketBroker contract on Arbitrum. You enter the deposit and reserve amounts in ETH.

{/* REVIEW: Rick — confirm Option 11 is still "Deposit broadcasting funds" in current builds */}

**Set max price per capability** (Option 16):

```
Menu option: "Set max price per capability"
```

Updates AI pipeline pricing (`-maxPricePerCapability`) at runtime for AI and dual gateways.

{/* REVIEW: Rick — confirm Option 16 is "Set max price per capability" in current builds */}

### Docker-Specific Usage

When running your gateway inside Docker, use the container hostname (not `localhost`) when invoking livepeer_cli:

```yaml
# docker-compose.yml
services:
  gateway:
    container_name: livepeer_gateway
    # ...
```

```bash
# Access livepeer_cli from inside the container
docker exec -it livepeer_gateway livepeer_cli -host livepeer_gateway -http 5935
```

If you use `localhost` inside Docker, the connection will fail because localhost inside the container refers to the container itself, not the running gateway process.

## CLI HTTP API

The livepeer_cli communicates with the node via an HTTP API on port 5935. You can call this API directly for automation, scripting, or integration with external monitoring.

```bash
# Get current node status
curl http://localhost:5935/status

# Get hardware stats
curl http://localhost:8935/hardware/stats
```

Full API reference: [CLI HTTP API Reference](/v2/gateways/resources/technical/api-reference/CLI-HTTP/cli-http-api)

## Which Tool When?

<AccordionGroup>

  <Accordion title="Is my gateway funded?">
    `livepeer_cli → Get node status` for the current balance, or Arbiscan to see the full deposit history and confirm on-chain state.
  </Accordion>

  <Accordion title="Are orchestrators redeeming my tickets?">
    Arbiscan → your gateway address → Internal Txns or Events → filter for `RedeemWinningTicket` events. Livepeer Explorer gives a summarised view.
  </Accordion>

  <Accordion title="Are AI jobs routing successfully?">
    Livepeer Tools for network-wide AI orchestrator status. Check `/health` on port 8937 and test a job directly if routing seems stuck.
  </Accordion>

  <Accordion title="What orchestrators are available for video?">
    Livepeer Explorer for the registered set with pricing and capacity. `livepeer_cli → List registered orchestrators` for what your node currently sees.
  </Accordion>

  <Accordion title="What's my GPU doing?">
    `curl http://localhost:8935/hardware/stats` — GPU utilisation, memory used/total, temperature.
  </Accordion>

  <Accordion title="Is my deposit draining faster than expected?">
    Arbiscan (redemption events + timestamp) to identify the pattern, then `livepeer_suggested_gas_price` in Prometheus to check if a gas spike caused orchestrators to redeem more aggressively.
  </Accordion>

  <Accordion title="I need historical metrics over time.">
    Prometheus + Grafana via the [Docker monitoring stack](/v2/gateways/guides/monitoring-and-tooling/monitoring-setup#docker-monitoring-stack). livepeer_cli and the HTTP API show current state only.
  </Accordion>

  <Accordion title="I need on-chain transaction history.">
    Arbiscan — it is the authoritative record of what happened on Arbitrum. Prometheus metrics represent what the node reported to itself, which may differ from on-chain truth if the node crashed or had an RPC gap.
  </Accordion>

</AccordionGroup>
