# Gateway Pricing Strategy

Your gateway sets a **maximum price cap** for each type of work it sends to orchestrators. Orchestrators advertise their own prices; your gateway accepts only those within your cap. Set the cap too low and no orchestrators are in range. Set it too high and you overpay. This guide explains how the pricing model works, how to configure it for video and AI workloads, and how to position competitively.

<Note>
  **Gateways pay orchestrators. They do not charge clients.** The flags on this page control what your gateway pays for compute. What you charge your application's end users is a separate concern implemented at your application layer.
</Note>

---

## Pricing fundamentals

All Livepeer payments are in **ETH (wei)**, regardless of the workload. There is no LPT in the payment flow.

Gateways set **caps**, not fixed prices. When your gateway selects an orchestrator from its list, it checks the orchestrator's advertised price against your cap. Any orchestrator above the cap is skipped for that job. If no orchestrator is within your cap, the job fails unless you have enabled `ignoreMaxPriceIfNeeded`.

This means your pricing configuration is really about **orchestrator access**: setting the range of the network you can afford to use.

---

## Video transcoding pricing

Video transcoding is priced **per pixel**. The gateway calculates fees based on the resolution and length of each transcoded segment.

<Card>
  **Key flags for video pricing**

  | Flag | Type | Default | Description |
  |---|---|---|---|
  | `-maxPricePerUnit` | int (wei) or USD string | `0` (any price) | Maximum price per `pixelsPerUnit` pixels |
  | `-pixelsPerUnit` | int | `1` | Granularity: how many pixels constitute one pricing unit |
  | `-ignoreMaxPriceIfNeeded` | bool | `false` | Accept any price if no orchestrators are in range |
</Card>

**How fee calculation works:** The effective price cap is `maxPricePerUnit` wei per `pixelsPerUnit` pixels. A 1920x1080 frame contains 2,073,600 pixels. With a cap of 1,000 wei per pixel and `pixelsPerUnit` of 1, that frame would cost at most 2,073,600 wei (about 0.000002 ETH).

Most operators leave `pixelsPerUnit` at the default of `1` and adjust only `maxPricePerUnit`.

**Example video configuration:**

```bash
livepeer \
  -gateway \
  -maxPricePerUnit 1000 \
  -pixelsPerUnit 1 \
  ...
```

### USD-denominated pricing

If you prefer to configure prices in USD rather than wei, you can pass a USD value with `-maxPricePerUnit`:

```bash
-maxPricePerUnit 0.02USD
```

This requires a **Chainlink ETH/USD price feed** to be configured via `-priceFeedAddr`. The gateway converts your USD cap to wei at runtime using the oracle price.

## AI pipeline pricing

AI workloads are priced differently from video. Each pipeline and model can have its own price cap, and the unit of measurement varies by pipeline type.

### Payment types for AI

Livepeer's AI payment system uses three distinct models depending on the pipeline:

| Payment type | Used for | Unit |
|---|---|---|
| **Per-pixel** | Image and video generation pipelines | `width × height × outputs` |
| **Per-request** | Single-output pipelines | One payment per API call |
| **Live video interval** | Live AI (live-video-to-video) | Time-based, every N seconds |

The `live-video-to-video` pipeline uses interval-based payments controlled by `-livePaymentInterval`. All other AI pipelines use per-pixel or per-request models based on their output type.

### Configuring AI price caps

Use `-maxPricePerCapability` to set per-pipeline caps as a JSON structure. This flag accepts either a JSON string directly or a path to a JSON file.

**Example `maxPrices.json`:**

```json
{
  "capabilities_prices": [
    {
      "pipeline": "text-to-image",
      "model_id": "default",
      "price_per_unit": 4768371,
      "pixels_per_unit": 1
    },
    {
      "pipeline": "image-to-image",
      "model_id": "ByteDance/SDXL-Lightning",
      "price_per_unit": 1700000,
      "pixels_per_unit": 1
    },
    {
      "pipeline": "image-to-video",
      "model_id": "default",
      "price_per_unit": 3390842,
      "pixels_per_unit": 1
    },
    {
      "pipeline": "audio-to-text",
      "model_id": "default",
      "price_per_unit": 12882811,
      "pixels_per_unit": 1
    },
    {
      "pipeline": "live-video-to-video",
      "model_id": "default",
      "price_per_unit": 100000,
      "pixels_per_unit": 1
    }
  ]
}
```

Pass it at startup:

```bash
-maxPricePerCapability /etc/livepeer/maxPrices.json
```

<Note>
  Use `"model_id": "default"` to apply a cap to all models in a pipeline. You can mix specific model caps with a default fallback in the same file: the specific model match takes precedence.
</Note>

**Pipeline-specific unit notes:**

- **Image and video pipelines** (text-to-image, image-to-image, image-to-video): price is per pixel (`width × height × outputs`).
- **Audio pipelines** (audio-to-text): price is per millisecond of audio.
- **LLM pipelines**: price is per token where supported. {/* REVIEW: Confirm LLM per-token pricing is implemented in go-livepeer. j0sh to verify. */}
- **Live video (live-video-to-video)**: interval-based; see below.

### Live AI interval pricing

For the `live-video-to-video` pipeline, your gateway sends periodic payments to the orchestrator rather than per-frame or per-request. The interval is configured with:

```bash
-livePaymentInterval 5s
```

The default is 5 seconds. Orchestrators receive a payment every interval for the duration of the live session. A shorter interval means more frequent, smaller payments; a longer interval means fewer, larger ones. Most operators use the default.

## All pricing flags at a glance

| Flag | Workload | Default | Notes |
|---|---|---|---|
| `-maxPricePerUnit` | Video + AI fallback | `0` (any price) | Sets global cap in wei or USD |
| `-pixelsPerUnit` | Video | `1` | Pixels per pricing unit |
| `-maxPricePerCapability` | AI | None | JSON file or string with per-pipeline caps |
| `-livePaymentInterval` | Live AI only | `5s` | Payment frequency for live-video-to-video |
| `-ignoreMaxPriceIfNeeded` | Both | `false` | Accept any price when no in-range orchestrators exist |
| `-priceFeedAddr` | Both (USD mode) | None | Chainlink oracle address for ETH/USD conversion |

## Competitive positioning

Your price cap determines which orchestrators your gateway can use. The Livepeer network is an open marketplace: orchestrators set their own prices based on their costs and target margins.

**Finding out what orchestrators charge:**

You can query orchestrator pricing directly via `livepeer_cli` or the Livepeer Explorer at [explorer.livepeer.org](https://explorer.livepeer.org). This gives you a baseline for setting competitive caps.

{/* REVIEW: Confirm whether livepeer_cli has a specific option for querying orchestrator pricing, and which option number it is. Rick / j0sh to verify. */}

**Orchestrators can also set per-gateway prices.** If an orchestrator wants to work with you specifically, they can configure a custom price for your gateway address using `-pricePerGateway`. This means your effective access to specific orchestrators may depend on relationships and reputation as well as your price cap.

**Dynamic price adjustment by orchestrators (`autoAdjustPrice`):** Some orchestrators enable automatic price adjustment based on current gas costs. This means the prices they advertise can shift. If your cap is tight, a gas spike can briefly knock all orchestrators out of range until prices stabilise.

### Setting a sensible starting cap

If you are starting fresh and have no baseline data, a practical approach is:

1. Query the Livepeer Explorer for the cheapest orchestrators currently active on the pipeline you need.
2. Set your cap 20 to 30 percent above their current rate. This gives you headroom during gas spikes without wildly overpaying.
3. Monitor job success rates. If jobs fail frequently, raise the cap. If all jobs succeed consistently, try lowering it gradually.

## Configuration methods

Pricing can be set at startup, in a config file, or updated at runtime without restarting the gateway.

<Tabs>
  <Tab title="CLI flags">
    Pass flags directly in your startup command:

    ```bash
    livepeer \
      -gateway \
      -maxPricePerUnit 5000000 \
      -pixelsPerUnit 1 \
      -maxPricePerCapability /etc/livepeer/maxPrices.json \
      -livePaymentInterval 5s \
      ...
    ```
  </Tab>

  <Tab title="Config file">
    Create a config file in key-value format and pass it with `-config`:

    ```
    maxPricePerUnit 5000000
    pixelsPerUnit 1
    maxPricePerCapability /etc/livepeer/maxPrices.json
    livePaymentInterval 5s
    ```

    ```bash
    livepeer -gateway -config /etc/livepeer/gateway.conf
    ```
  </Tab>

  <Tab title="HTTP API (runtime)">
    Adjust pricing without restarting the gateway via the internal HTTP API:

    ```bash
    # Set global broadcast config (video pricing)
    curl -X POST http://127.0.0.1:5935/setBroadcastConfig \
      -d "maxPricePerUnit=5000000&pixelsPerUnit=1"

    # Set AI pipeline cap at runtime
    curl -X POST http://127.0.0.1:5935/setMaxPriceForCapability \
      -d '{"pipeline":"text-to-image","model_id":"default","price_per_unit":4768371,"pixels_per_unit":1}'
    ```
