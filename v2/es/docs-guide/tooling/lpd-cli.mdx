<Tip>
`lpd` is the repository CLI for setup, local development, test orchestration, hook management, and script execution.
Use this page as both your first-run guide and full command/flag reference.
</Tip>

## ¿Qué es lpd

`lpd` estandariza la configuración y operaciones locales de docs para este repositorio.
Envuelve la configuración, la vista previa, las pruebas, los ganchos y los scripts gestionados detrás de una sola superficie de comando.
La fuente de verdad del comportamiento es el repositorio-raíz [`lpd`](/lpd) script.

## Empieza aquí (Camino para nuevos colaboradores)

1. Instala/arranca el entorno CLI del repositorio.
2. Verifica tu entorno y estado de ganchos.
3. Inicia la vista previa local de docs.
4. Ejecuta verificaciones básicas antes de abrir una PR.

## Instalación

Comando estándar de instalación/arranque:

```bash
bash lpd setup --yes
```

¿Qué `setup` hace?

- Instala `tools` dependencias (`cd tools && npm install`)
- Instala `tests` dependencias (`cd tests && npm install`)
- Instala/actualiza ganchos de git (`.githooks/install.sh`)
- Instala opcionalmente `lpd` en tu PATH (a menos que `--no-cli`)

Verificar instalación:

```bash
lpd version
lpd doctor
```

Si el PATH aún no se ha actualizado:

```bash
bash lpd version
bash lpd doctor
```

Variantes comunes:

```bash
lpd setup --no-cli
lpd setup --no-tests
lpd setup --with-mintlify
LPD_INSTALL_DIR="$HOME/.local/bin" lpd setup --yes
```

<Note>
If `lpd setup` updates your shell rc file, open a new terminal or run `source ~/.zshrc` (or your active shell rc) before using plain `lpd`.
</Note>

## Flujo de primeras 10 minutos

### Solo necesito docs en ejecución

```bash
bash lpd setup --yes
lpd dev -- --port 3333
```

### Necesito ejecutar comprobaciones antes de PR

```bash
lpd doctor --strict
lpd test --staged
lpd ci --skip-browser
```

### Necesito ejecutar un script gestionado

```bash
lpd scripts list --group tools
lpd tools dev test-add-callouts -- --help
```

    </ResponseField>
    <ResponseField name="info" type="subcommand">

    </ResponseField>
    <ResponseField name="<hook-script tokens...>" type="subcommand">
      Fallback path: unresolved hooks subcommands are treated as managed script tokens in `.githooks/`.
    </ResponseField>
  </ResponseField>

    Resolve and execute a managed script command in a script group.

    **Usage**

    **Flags**
    <ResponseField name="--dry-run" type="boolean" default="false">

    </ResponseField>
    <ResponseField name="--yes" type="boolean" default="false">

    </ResponseField>
    <ResponseField name="--json" type="boolean" default="false">

    </ResponseField>

    Group shorthand alias for `lpd scripts run <group> ...`.

    **Usage**
    ```bash
    lpd tools <script-path tokens...> [-- script args]
    lpd tasks <script-path tokens...> [-- script args]
    lpd tests <script-path tokens...> [-- script args]
    lpd v2 <script-path tokens...> [-- script args]
    ```

    **Examples**
    ```bash
    lpd tools dev test-add-callouts -- --help
    lpd v2 wcag-repair-common -- --staged --stage
    ```
  </ResponseField>

  <ResponseField name="Script risk model" type="command">
    Scripts are assigned risk by group and may require confirmation.

    **Usage**
    ```bash
    low: tests
    medium: tools, hooks
    high: tasks, v2
    ```

    **Examples**
    ```bash
    lpd scripts run tasks run-audit
    lpd scripts run tasks run-audit --yes
    ```

    High-risk groups prompt for confirmation unless `--yes` is supplied.
    In non-interactive mode, high-risk commands fail without `--yes`.
  </ResponseField>
</Accordion>

## Solución de problemas

### `lpd` no encontrado

Usa el respaldo local del repositorio y verifica la configuración:

```bash
bash lpd setup --yes
bash lpd version
```

Si es necesario, recarga tu archivo rc de shell (`source ~/.zshrc` o equivalente).

### Ganchos no coinciden

Comprueba e reinstala los ganchos:

```bash
lpd hooks status
lpd hooks install
lpd hooks verify
```

### `--json` expectativas

El modo JSON global es forma de prefijo: `lpd --json <command> ...`.
Algunos comandos también aceptan local `--json`.
`lpd info --json` no es válido porque `info` solo acepta `--help|-h`.

### Colisión de puerto en `lpd dev`

Pasa un puerto a través del lanzador:

```bash
lpd dev -- --port 3334
```

## Salida JSON

Cuando el modo JSON está activo en comandos basados en finalización, la salida utiliza este sobre de estado:

<Accordion title="JSON Output Envelope" icon="code">
  <ResponseField name="ok" type="boolean">
    `true` for success and `false` for failure.
  </ResponseField>

  <ResponseField name="command" type="string">
    Normalised command label (for example: `test`, `scripts list`, `hooks status`).
  </ResponseField>

  <ResponseField name="message" type="string">
    Human-readable summary of the command result.
  </ResponseField>

  <ResponseField name="exit_code" type="number">
    Exit code returned by the command path.
  </ResponseField>

  <ResponseField name="repo_root" type="string">
    Absolute repo root used during command execution.
  </ResponseField>

  Example:

  ```json
  {
    "ok": true,
    "command": "version",
    "message": "lpd 0.2.0",
    "exit_code": 0,
    "repo_root": "/path/to/livepeer-docs-repo"
  }
  ```

  <Note>
  `lpd info` does not emit a structured JSON envelope.
  Use JSON mode with commands that complete via `finish()`.
  </Note>

</Accordion>
