A pool lets you contribute GPU compute to the Livepeer network without running a full orchestrator yourself. The pool operator handles staking, on-chain registration, reward calling, and routing — you provide the GPU, you earn for the work it processes.

This is the **lowest-barrier entry point** for GPU owners who want to monetise compute without taking on protocol operations.

<Note>
  Pools are community-run. They are not an official Livepeer Foundation product and each pool operates independently. Vet your pool carefully before connecting your hardware.
</Note>

---

## How pools work

An orchestrator pool is a single registered orchestrator node that aggregates compute capacity from multiple GPU contributors. From the network's perspective, it is one orchestrator with one stake. From your perspective, your GPU is one worker behind that orchestrator.

The pool operator:
- Holds and manages the stake
- Sets prices and advertises capabilities
- Receives work from gateways
- Routes segments to your GPU for processing
- Distributes payouts to pool members

You:
- Run a transcoding worker (a go-livepeer process in worker mode, or the pool operator's own client)
- Process the segments routed to your GPU
- Receive off-chain payouts based on the work your GPU completed

**There is no on-chain record of your individual contribution.** Payouts are made by the pool operator directly, usually in ETH, and sometimes in pool-specific tokens.

## Pool vs solo orchestrator

| | Pool worker | Solo orchestrator |
|---|---|---|
| LPT stake required | No | Yes |
| On-chain registration | No (pool handles this) | Yes |
| Set your own prices | No | Yes |
| Direct on-chain payouts | No (via pool operator) | Yes |
| Ops complexity | Low | Medium–High |
| Trust requirement | Pool operator | Yourself |
| Reward calling responsibility | Pool operator | You (or Siphon) |
| Good for | GPU owners wanting passive income | Operators wanting full control |

### What to check before joining any pool

Not all pools are transparent about their terms. Before connecting your hardware, find out:

<AccordionGroup>

<Accordion title="How are payouts calculated?">
Reputable pools publish their payout formula. Some pay per segment transcoded; others pay proportionally to GPU time contributed. Ask for the formula and compare it to running a solo node.

Common models:
- **Per-segment** — you earn a fixed amount per segment processed
- **Proportional share** — your share of pool earnings matches your share of total work processed
- **Pool token** — some pools issue a native token (e.g. Titan Node issues LTNT) that can be exchanged for ETH or held

Always ask what you actually get paid *before* committing hardware.
</Accordion>

<Accordion title="What is the payout frequency and minimum threshold?">
Most pools pay weekly or monthly. There is usually a minimum balance before you receive a payout. Ask the pool operator what the threshold is and how long it typically takes for a new contributor to reach it.

If the minimum is high and network demand is low, it may take a long time before your first payout. Factor this into your decision.
</Accordion>

<Accordion title="What do they require from you?">
Legitimate pools give you clear technical requirements: GPU type, minimum VRAM, operating system, network bandwidth. They do not ask for your keystore or private key. **If a pool requests your keystore or private key at any point, that is a scam.**
</Accordion>

<Accordion title="Is the pool actively maintained?">
Check when the pool's GitHub repo or documentation was last updated. Check their Discord presence and community activity. A pool that has not been maintained in over a year may have stale tooling or may no longer be receiving meaningful work from gateways.
</Accordion>

</AccordionGroup>

## Step 2: Connect your GPU

Pool operators each have their own worker client setup. The Titan Node pool uses a purpose-built client. The general connection model is consistent across pools:

### Option A: Docker worker (recommended)

Most pools provide a Docker image. This is the simplest path — you do not need to compile go-livepeer yourself.

```bash
# Clone the pool's repository
git clone https://github.com/Titan-Node/Titan-Node-Pool.git
cd Titan-Node-Pool/docker

# Review and edit the docker-compose.yml with your settings
# Then start the worker
docker compose up -d
```

Your container connects to the pool operator's orchestrator, receives video segments, processes them with your GPU, and reports back completed work.

<Card title="Titan Node Pool Docker README" icon="github" href="https://github.com/Titan-Node/Titan-Node-Pool/tree/main/docker">
  Docker Compose configuration and setup steps for connecting to the Titan pool.
</Card>

### Option B: go-livepeer in transcoder mode

If you prefer to run go-livepeer directly rather than via a Docker image, run it in transcoder mode and point it at the pool operator's orchestrator address:

```bash
livepeer \
    -transcoder \
    -orchAddr <pool-orchestrator-address>:8935 \
    -nvidia 0 \
    -maxSessions 10
```

The pool operator will provide their orchestrator address and any authentication configuration.

<Warning>
  Do not pass `-orchestrator` when joining a pool — that would attempt to register you as a new orchestrator on-chain. You want `-transcoder` mode only. In transcoder mode your machine processes work; it does not register independently on the network.
</Warning>

### Option C: Cloud GPU

Some operators contribute cloud GPU instances (RunPod, Lambda Labs, AWS) as pool workers. If you do this, your compute cost must be lower than your expected earnings. Check current network demand in Discord before committing to a cloud GPU rental.

## Step 3: Verify you are receiving work

Once connected, you should see transcoding activity in your worker logs within a few minutes if the pool is active and routing work to you.

**In go-livepeer transcoder mode, look for:**

```
Received transcode request  segment=0 ...
Transcoded segment          segment=0 ...
```

**If you see no activity after 10–15 minutes:**

1. Verify the pool orchestrator endpoint address is correct
2. Check your GPU is visible: `nvidia-smi` should show the GPU, and go-livepeer should log a GPU detection line at startup
3. Check that your port 8935 is reachable from the pool orchestrator (some pools need inbound connectivity)
4. Ask in the pool's Discord channel — pools sometimes have quiet periods when network demand is low

---

## Step 4: Track your earnings

Pool payouts are off-chain. Your earnings are not visible on the Livepeer Explorer the way solo orchestrator fees are. Each pool has its own tracking mechanism:

- **Titan Node Pool:** public dashboard at `app.titan-node.com` shows per-worker stats and pending payout
- **Most pools:** Discord bot or web dashboard showing your worker's segment count and balance

Keep a record of expected vs actual payouts. If you believe you are not being paid correctly, raise it in the pool's community channel.

## Frequently asked questions

<AccordionGroup>

<Accordion title="Do I need LPT to join a pool?">
No. Staking LPT is the pool operator's responsibility. As a worker you contribute compute only — no token purchase or on-chain action is required from you.
</Accordion>

<Accordion title="Is my GPU identified on-chain?">
No. On-chain, only the orchestrator (the pool node) is registered. Your individual contribution is tracked only within the pool operator's own systems and any dashboard they provide.
</Accordion>

<Accordion title="Can I earn LPT as well as ETH through a pool?">
Transcoding fees are paid in ETH. LPT inflation rewards accrue to the staking orchestrator. Some pools choose to share a portion of LPT rewards with workers — ask your pool operator about their specific policy.
</Accordion>

<Accordion title="Can I run AI inference workloads through a pool?">
The AI subnet requires capability registration coupled to the orchestrator node. As of early 2026, publicly documented pools primarily handle video transcoding. If you specifically want to run AI inference workloads and earn from them, a solo orchestrator is currently the more viable path. Check Discord for any emerging AI pools.
</Accordion>

<Accordion title="Can I contribute multiple GPUs?">
Yes. Pass multiple GPU IDs when starting your worker (e.g. `-nvidia 0,1,2`). The pool orchestrator sees your worker as a single endpoint with higher capacity. Most pools support this without any special configuration.
</Accordion>

<Accordion title="How does this compare to other GPU mining pools?">
Livepeer transcoding pools work differently from proof-of-work mining pools — there is no hashrate, and earnings are directly tied to actual video segments processed. This means earnings are demand-dependent rather than fixed, and there is no hardware arms race in the same way as PoW. The primary variable is network demand for transcoding, which varies with how many applications are routing video through the Livepeer network.
</Accordion>

</AccordionGroup>
