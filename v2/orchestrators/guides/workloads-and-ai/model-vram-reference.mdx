GPU memory (VRAM) is the primary constraint for AI inference operators on Livepeer. The models you can run, how many pipelines you can warm simultaneously, and how competitive you are on latency all depend on how you allocate VRAM across your workloads.

This page is a practical operator reference: VRAM figures per pipeline, warm model strategy, multi-GPU patterns, and complete `aiModels.json` field documentation.

---

## VRAM by pipeline

## GPU reference by persona

### Consumer GPU tier (8â€“12 GB VRAM)

RTX 2060 (6 GB), RTX 3060 12 GB, RTX 2060 Super (8 GB), RTX 3060 Ti (8 GB)

**Viable pipelines:**
- `llm` â€” Llama 8B Q4 via Ollama (~6â€“8 GB)
- `image-to-text` â€” BLIP large (~2 GB)
- `audio-to-text` â€” Whisper large-v3 (~3 GB) âś… fits on 8 GB cards
- `segment-anything-2` â€” SAM2 base model (~4â€“6 GB)

**Not viable:** `text-to-image`, `image-to-image`, `image-to-video`, `live-video-to-video`

**Strategy for this tier:** Run `audio-to-text` and `image-to-text` warm simultaneously â€” both fit easily in 8 GB and together give you two income streams. Add `llm` on a separate GPU if available.

---

### Mid tier (16â€“20 GB VRAM)

RTX 3090 (24 GB, but close in practice with overheads), RTX 3080 Ti (12 GB â€” insufficient), A5000 (24 GB)

**Note:** The effective threshold for diffusion pipelines is 24 GB. A nominal 16 GB card leaves insufficient headroom for SDXL warm + inference peaks.

**Viable pipelines on 24 GB:**
- `text-to-image` â€” SDXL Lightning warm âś…
- `image-to-image` â€” SDXL Lightning cold (warm clashes with text-to-image)
- `audio-to-text` â€” Whisper warm simultaneously (only 3 GB â€” fits alongside diffusion)
- `upscale` â€” SD x4 upscaler warm

**Strategy:** Warm `text-to-image` as the primary pipeline. Co-warm `audio-to-text` as a secondary â€” they barely overlap in VRAM. Cold-load `image-to-image` on demand.

---

### High tier (24 GB+ per GPU, multiple GPUs)

RTX 4090 (24 GB), A100 40/80 GB, H100 80 GB

**Full pipeline access.** With multiple GPUs:
- RTX 4090 Ă— 2: warm SDXL on GPU0, warm Whisper + LLM on GPU1
- A100 80 GB: can warm multiple diffusion models + run `live-video-to-video` simultaneously
- H100: full fleet for multi-stream real-time AI at production scale

## Warm vs cold â€” when it matters

### Pipelines where warm is competitively critical

- **`text-to-image`** â€” cold load on SDXL takes 30â€“90 seconds. Gateways will route to warm competitors first. Running cold is essentially not participating.
- **`live-video-to-video`** â€” cold loading mid-stream causes noticeable interruption. Always warm for real-time pipelines.
- **`image-to-image`** â€” competitive warm advantage, though less severe than text-to-image.

### Pipelines where cold loading is acceptable

- **`audio-to-text`** â€” Whisper loads in ~3â€“5 seconds. First-request latency is tolerable for transcription use cases.
- **`image-to-text`** â€” BLIP is very fast to load. Cold loading is acceptable.
- **`segment-anything-2`** â€” depends on use case; applications requiring real-time segmentation need warm.

### The Beta warm model constraint

During the Beta phase, only **one warm model per GPU** is supported. If you set `warm: true` on more `aiModels.json` entries than you have GPUs, the AI worker logs a conflict at startup and skips excess warm entries.

**Example of what works:**

One RTX 4090, two entries â€” only one can be warm:

```json
[
  {
    "pipeline": "text-to-image",
    "model_id": "SG161222/RealVisXL_V4.0_Lightning",
    "price_per_unit": 4768371,
    "warm": true
  },
  {
    "pipeline": "audio-to-text",
    "model_id": "openai/whisper-large-v3",
    "price_per_unit": 12882811
  }
]
```

`text-to-image` is warm (your primary revenue pipeline). `audio-to-text` loads cold on demand.

**Exception:** Whisper is small enough (3 GB) that co-warming alongside a diffusion model is sometimes possible without conflict â€” but this is hardware-dependent and not guaranteed. Monitor startup logs.

## Multi-GPU configuration

The AI worker assigns GPU resources based on the order of entries in `aiModels.json` and available device IDs. For explicit multi-GPU assignment, use the CUDA device environment variable approach when launching containers:

```bash
# GPU 0 handles diffusion pipelines
docker run -d --gpus '"device=0"' --name ai-runner-gpu0 ...

# GPU 1 handles Whisper and LLM
docker run -d --gpus '"device=1"' --name ai-runner-gpu1 ...
```

For a node with RTX 4090 (GPU0) + RTX 2060 (GPU1):

```json
[
  {
    "pipeline": "text-to-image",
    "model_id": "SG161222/RealVisXL_V4.0_Lightning",
    "price_per_unit": 4768371,
    "warm": true
  },
  {
    "pipeline": "image-to-image",
    "model_id": "ByteDance/SDXL-Lightning",
    "price_per_unit": 4768371
  },
  {
    "pipeline": "audio-to-text",
    "model_id": "openai/whisper-large-v3",
    "price_per_unit": 12882811,
    "warm": true,
    "url": "http://whisper-runner:8001"
  },
  {
    "pipeline": "llm",
    "model_id": "meta-llama/Meta-Llama-3.1-8B-Instruct",
    "warm": true,
    "price_per_unit": 0.18,
    "currency": "USD",
    "pixels_per_unit": 1000000,
    "url": "http://llm_runner:8000"
  }
]
```

In this configuration: GPU0 handles both diffusion pipelines (warm text-to-image, cold image-to-image). GPU1 handles Whisper and LLM via external containers on its own VRAM.

## aiModels.json complete schema

Every field, its type, valid values, and behaviour:

<AccordionGroup>

<Accordion title="pipeline (required)">
**Type:** string
**Description:** The pipeline identifier. Must exactly match a supported pipeline name.

**Valid values:**
```
text-to-image
image-to-image
image-to-video
image-to-text
audio-to-text
segment-anything-2
text-to-speech
upscale
llm
live-video-to-video
```
</Accordion>

<Accordion title="model_id (required)">
**Type:** string
**Description:** HuggingFace model ID. Case-sensitive. Must include the organisation prefix.

**Examples:**
- `"SG161222/RealVisXL_V4.0_Lightning"` âś…
- `"RealVisXL_V4.0_Lightning"` âťŚ (missing org prefix)
- `"sg161222/realvisxl_v4.0_lightning"` âťŚ (wrong case)
- `"llama3.1:8b"` âťŚ (Ollama tag, not HuggingFace ID)

For `llm` pipeline, use the HuggingFace ID (`meta-llama/Meta-Llama-3.1-8B-Instruct`) even though the Ollama runner uses its own internal tag format.
</Accordion>

<Accordion title="price_per_unit (required)">
**Type:** integer or string
**Description:** Price per unit of work.

**Integer format** â€” value in Wei:
```json
"price_per_unit": 4768371
```

**USD string format** â€” scientific notation with USD suffix:
```json
"price_per_unit": "0.5e-3USD",
"currency": "USD"
```

When using USD notation, also set `"currency": "USD"`.

**Pricing unit per pipeline:**

| Pipeline | Unit |
|---|---|
| `text-to-image` | Per output pixel (width Ă— height) |
| `image-to-image` | Per output pixel |
| `image-to-video` | Per output pixel |
| `image-to-text` | Per input pixel |
| `audio-to-text` | Per millisecond of audio |
| `upscale` | Per input pixel |
| `llm` | Custom (typically per 1M tokens) |
| `live-video-to-video` | Per frame |
</Accordion>

<Accordion title="warm (optional)">
**Type:** boolean
**Default:** false
**Description:** Preload the model into GPU VRAM at container startup.

Setting `warm: true` eliminates cold-start latency. One warm model per GPU during Beta.

If the model is too large for available VRAM and `warm` is true, the container will fail at model load time. Check `docker logs` for OOM messages.
</Accordion>

<Accordion title="pixels_per_unit (optional)">
**Type:** integer
**Description:** Units of work per pricing unit. Adjusts the effective per-unit cost granularity.

Used primarily with `audio-to-text` to set per-millisecond pricing:
```json
"pixels_per_unit": 1
```

For `llm`, sets the token count per pricing unit:
```json
"pixels_per_unit": 1000000
```
(1 pricing unit = 1 million tokens)
</Accordion>

<Accordion title="currency (optional)">
**Type:** string
**Description:** Currency for `price_per_unit`. Required when using USD notation.

```json
"currency": "USD"
```
</Accordion>

<Accordion title="url (optional)">
**Type:** string
**Description:** URL of an external container serving this pipeline. The AI worker treats the URL as a pass-through and polls `/health` at startup.

Use this for:
- Ollama LLM runner
- Custom inference servers
- K8s clusters or GPU farms

```json
"url": "http://llm_runner:8000"
```

The container at this URL must:
1. Expose `/health` â†’ return HTTP 200
2. Handle inference requests in the format the AI worker sends
</Accordion>

<Accordion title="token (optional)">
**Type:** string
**Description:** Bearer token for authenticating with the external container.

```json
"token": "my-secret-token"
```
</Accordion>

<Accordion title="capacity (optional)">
**Type:** integer
**Default:** 1
**Description:** Maximum concurrent inference tasks from this container. Set based on the container's actual concurrency support.

```json
"capacity": 2
```

For Ollama with multiple loaded models or hardware that supports parallel inference, increasing capacity can improve throughput.
</Accordion>

<Accordion title="optimization_flags (optional)">
**Type:** object
**Description:** Performance optimisations for warm diffusion models. Applies to `text-to-image`, `image-to-image`, and `upscale` pipelines with `warm: true`.

**SFAST** (Stable Fast, up to 25% speedup, no quality loss):
```json
"optimization_flags": { "SFAST": true }
```

**DEEPCACHE** (up to 50% speedup, minor quality impact):
```json
"optimization_flags": { "DEEPCACHE": true }
```

**Do not combine.** Do not use DEEPCACHE with Lightning or Turbo models (they are already step-optimised).

**Sources:** [Stable Fast](https://github.com/chengzeyi/stable-fast) Â· [DeepCache](https://github.com/horseee/DeepCache)
</Accordion>

</AccordionGroup>

## Model selection and earnings

Not all models earn equally. Earnings depend on:

1. **Pipeline demand** â€” some pipelines receive more jobs from gateways
2. **Model match** â€” gateways often request specific model IDs; running the requested model gets you the job
3. **Warm status** â€” warm models win latency-competitive pipelines
4. **Pricing competitiveness** â€” prices above the gateway's `maxPricePerUnit` receive zero jobs

### Tracking performance

- **[tools.livepeer.cloud/ai/network-capabilities](https://tools.livepeer.cloud/ai/network-capabilities)** â€” real-time capability visibility across all orchestrators
- **[explorer.livepeer.org](https://explorer.livepeer.org)** â€” per-orchestrator fee earnings and job count

Use the Explorer AI leaderboard to compare your earnings against similar nodes. Nodes with identical hardware but different warm model selections often show significantly different earnings â€” the warm pipeline choice matters more than raw GPU performance.

### Model selection heuristics

| Goal | Model choice |
|---|---|
| Maximum text-to-image jobs | `SG161222/RealVisXL_V4.0_Lightning` warm |
| Entry with 8 GB GPU | `openai/whisper-large-v3` + `Salesforce/blip-image-captioning-large` |
| LLM income with older card | `meta-llama/Meta-Llama-3.1-8B-Instruct` via Ollama |
| Real-time AI premium jobs | StreamDiffusion via ComfyStream |
| Diversified single-GPU (24 GB) | `text-to-image` warm, `audio-to-text` cold, `image-to-image` cold |

## Pricing strategy

### Understanding the market

Prices are set by operators and enforced at the gateway layer. If your `price_per_unit` is higher than a gateway's `maxPricePerUnit`, you receive no jobs from that gateway â€” regardless of how performant your node is.

The network is price-competitive. Setting prices too high means no jobs. Setting prices too low reduces earnings unnecessarily.

These figures are approximate and shift with ETH/USD rates and network competition. Use [explorer.livepeer.org](https://explorer.livepeer.org) for current market rates.

## Hosting custom models (BYOC)

The `url` field in `aiModels.json` allows any inference server to serve a pipeline â€” not just the standard `livepeer/ai-runner` containers. This is the BYOC (Bring Your Own Container) path.

BYOC use cases:
- Running models not available on HuggingFace
- Fine-tuned proprietary models
- Custom inference architectures (TensorRT, ONNX, OpenVINO)
- Models hosted in K8s or a GPU farm behind a load balancer

The only requirement is the `/health` endpoint contract and matching the AI worker's request format. See [Hosting Models (BYOC)](/v2/orchestrators/guides/advanced-operations/large-scale-operations) for the full BYOC guide.
