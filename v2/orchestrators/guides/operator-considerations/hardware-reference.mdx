Hardware is the constraint for every workload type on Livepeer. This page is a practical reference: GPU support policy, session limits by card, VRAM tiers by pipeline, and the minimum system stack needed before your node goes live.

<CustomDivider />

## GPU vendor support

**NVIDIA is the supported GPU vendor for go-livepeer.** AMD and Intel GPUs are not supported for hardware-accelerated transcoding or AI inference. NVIDIA CUDA is required for both NVENC video transcoding and all AI inference pipelines.

### Driver requirements

Before anything else, ensure your NVIDIA drivers are up to date:

```bash icon="terminal"
# Check current driver version
nvidia-smi

# Output shows driver version and CUDA version
# e.g. NVIDIA-SMI 550.54.14    Driver Version: 550.54.14    CUDA Version: 12.4
```

If `nvidia-smi` fails or is not installed, install NVIDIA drivers for your OS before proceeding. go-livepeer cannot use your GPU without working NVIDIA drivers.

**Minimum CUDA version:** 12.0. Earlier versions are not supported by the AI runner containers.

<CustomDivider />

## NVENC/NVDEC session limits

NVIDIA consumer GPUs impose artificial **concurrent encoding session limits** via NVENC (hardware video encoder) and NVDEC (hardware video decoder). These limits directly constrain how many simultaneous transcoding streams your card can handle in hardware mode.

<Warning>
  Consumer NVIDIA cards (GeForce RTX series) are limited to a small number of concurrent NVENC sessions by default. Data centre cards (A-series, L-series, H-series) have no such cap.
</Warning>

Consumer cards are typically limited to **2–3 concurrent NVENC encode sessions** before they throttle or fail with `NvEncOpenEncodeSessionEx` errors. This is a hardware-enforced licensing restriction, not a software limitation in go-livepeer.

**Check the official NVIDIA matrix for your card's exact limits:**

→ [developer.nvidia.com/video-encode-decode-gpu-support-matrix](https://developer.nvidia.com/video-encode-decode-gpu-support-matrix)

## Hardware tiers by workload

### Video transcoding

Video transcoding is primarily NVENC/NVDEC throughput and memory bandwidth. It does **not** require large VRAM – model loading is not involved.

<StyledTable variant="bordered">
  <TableRow header>
    <TableCell header>Hardware tier</TableCell>
    <TableCell header>Example GPUs</TableCell>
    <TableCell header>Transcoding suitability</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>Consumer (NVENC-capped)</TableCell>
    <TableCell>RTX 3060, 3070, 3080, 4070, 4080</TableCell>
    <TableCell>2–3 concurrent sessions (NVENC cap) unless using unofficial patches – read NVIDIA ToS</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>Data centre / workstation</TableCell>
    <TableCell>A4000, A5000, RTX 6000, L4</TableCell>
    <TableCell>Uncapped – 10–20+ concurrent sessions depending on compute</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>High-end data centre</TableCell>
    <TableCell>A100, H100</TableCell>
    <TableCell>Uncapped – full concurrent session headroom</TableCell>
  </TableRow>
</StyledTable>

**Transcoding without a GPU:** Possible using CPU-only mode, but throughput is significantly lower. Practical capacity is usually low single-digit concurrent sessions and depends heavily on CPU generation, codec mix, and rendition ladder. Not competitive at scale.

### Batch AI inference

AI inference requires VRAM. The table below is a summary – full pipeline-by-pipeline VRAM figures are in [Models and VRAM Reference](/v2/orchestrators/guides/workloads-and-ai/model-vram-reference).

<StyledTable variant="bordered">
  <TableRow header>
    <TableCell header>VRAM tier</TableCell>
    <TableCell header>GPU examples</TableCell>
    <TableCell header>What you can run</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>**8–12 GB**</TableCell>
    <TableCell>RTX 3060 12GB, RTX 2060 Super 8GB, RTX 3060 Ti 8GB</TableCell>
    <TableCell>`audio-to-text` (Whisper), `image-to-text` (BLIP), `llm` (Llama 8B via Ollama), `segment-anything-2` (base)</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>**24 GB**</TableCell>
    <TableCell>RTX 3090, RTX 4090, A5000</TableCell>
    <TableCell>All diffusion pipelines (`text-to-image`, `image-to-image`, `image-to-video`, `upscale`), full model access</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>**40–80 GB**</TableCell>
    <TableCell>A100 40GB, A100 80GB, H100</TableCell>
    <TableCell>Multi-model warm, large LLMs (70B+), `image-to-video` at high quality, multiple simultaneous AI pipelines</TableCell>
  </TableRow>
</StyledTable>

The key threshold is **24 GB** for competitive diffusion pipelines. Below that, you cannot warm an SDXL model. Above it, you have full access to the most-requested AI pipelines on the network.

### Real-time AI inference (Cascade / live-video-to-video)

Real-time video AI has stricter requirements than batch AI:

- **Minimum VRAM:** 12–16 GB (StreamDiffusion with SD 1.5)
- **Competitive VRAM:** 24 GB (StreamDiffusion with SDXL)
- **Additional requirements:** Fast PCIe bandwidth for frame buffer throughput, low-latency memory
- **Frame buffer overhead:** Add 1–2 GB on top of the model's base VRAM footprint

Real-time pipelines need models warm at all times. A cold start during an active stream causes visible interruption. See [Realtime AI Setup](/v2/orchestrators/guides/workloads-and-ai/realtime-ai-setup) for the complete runtime requirements.

**Storage note:** AI model weights are large. A single SDXL model is approximately 7–8 GB. Running multiple pipelines requires fast storage (NVMe) to load models without introducing latency. Budget at least 100 GB of fast storage per 4–5 models you plan to serve.

**Network note:** Latency to gateways matters more than raw throughput for AI jobs. A 100 Mbps connection with latency under 20 ms to major regions outperforms a 1 Gbps connection with high latency for real-time AI workloads.

## Pre-launch checklist

Run through this before activating on the network:

<StyledSteps>

<StyledStep title="Verify GPU visibility">
```bash icon="terminal"
nvidia-smi
# Should output: GPU name, driver version, CUDA version, memory
# If this fails: drivers are not installed correctly
```
</StyledStep>

<StyledStep title="Verify Docker can see your GPU">
```bash icon="terminal"
docker run --rm --gpus all nvidia/cuda:12.0-base nvidia-smi
# Should output the same as nvidia-smi above
# If this fails: NVIDIA Container Toolkit is not installed or configured
```
</StyledStep>

<StyledStep title="Install NVIDIA Container Toolkit if needed">
```bash icon="terminal"
# Ubuntu 22.04 / 24.04
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
sudo apt-get update && sudo apt-get install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
```
See the [NVIDIA Container Toolkit install guide](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/install-guide.html) for other distributions.
</StyledStep>

<StyledStep title="Check required ports are open">
```bash icon="terminal"
# Default go-livepeer ports
# 8935: Orchestrator RPC
# 7935: CLI HTTP interface (localhost only)

# Check with:
sudo ufw status
# or
sudo iptables -L
```

Port 8935 must be reachable from the internet (or from your gateway if running a private setup). If behind NAT, configure port forwarding.
</StyledStep>

<StyledStep title="Verify Arbitrum RPC access">
Ensure your node has access to an Arbitrum One RPC endpoint. You can use:
- Public: `https://arb1.arbitrum.io/rpc`
- Alchemy, Infura, QuickNode (more reliable for production)

```bash icon="terminal"
# Quick test
curl -s https://arb1.arbitrum.io/rpc \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"eth_blockNumber","params":[],"id":1}' | python3 -m json.tool
```
</StyledStep>

<StyledStep title="Check static IP or configure DDNS">
Your `serviceAddr` (the address gateways use to reach you) should be stable. Home connections with dynamic IPs need DDNS or a static IP configured via your ISP. Changing your IP without updating `serviceAddr` means gateways lose contact.
</StyledStep>

<StyledStep title="Confirm ETH balance on Arbitrum">
Your orchestrator wallet needs ETH on Arbitrum before activation. Budget at least 0.01 ETH. See [Payments](/v2/orchestrators/guides/staking-and-rewards/payments) for ongoing ETH requirements.
</StyledStep>

</StyledSteps>
