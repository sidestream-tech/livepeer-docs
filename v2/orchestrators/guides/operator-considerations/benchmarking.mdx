Before you set a session limit and go live, run `livepeer_bench`. It simulates the real network workload — segments arriving in real time, multiple concurrent sessions — and tells you exactly how many streams your GPU can handle without falling behind.

This page covers: downloading the benchmark assets, running the tool, reading the output, and using the session scaling script to find your hardware ceiling.

<Note>
  Benchmarking only measures local transcoding capacity. It does not simulate network bandwidth, ticket redemption, or AI inference workloads. For session limit calculation that factors in both hardware and bandwidth, continue to [Bandwidth and Session Limits](/v2/orchestrators/guides/feasibility-and-hardware/session-limits) after completing this guide.
</Note>

<CustomDivider />

## What livepeer_bench does

`livepeer_bench` ships as part of go-livepeer. It:

- Takes an HLS stream and a set of output rendition profiles as input
- Simulates live mode by default — queuing segments one-by-one as they arrive, just as a real gateway would send them
- Runs multiple concurrent transcoding sessions (configurable)
- Outputs per-segment metrics as CSV and a summary table

The key output metric is the **Real-Time Duration Ratio**: total transcoding time divided by total source duration. A ratio below 1.0 means you're transcoding faster than real time. A ratio above 1.0 means your GPU is falling behind.

## Running the benchmark

### Basic run

```bash icon="terminal"
livepeer_bench \
    -in bbb/source.m3u8 \
    -transcodingOptions transcodingOptions.json \
    -nvidia 0 \
    -concurrentSessions 1
```

**Flag reference:**

<StyledTable variant="bordered">
  <TableRow header>
    <TableCell header>Flag</TableCell>
    <TableCell header>Description</TableCell>
    <TableCell header>Default</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>`-in`</TableCell>
    <TableCell>Path to the source `.m3u8` manifest</TableCell>
    <TableCell>Required</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>`-transcodingOptions`</TableCell>
    <TableCell>Path to JSON rendition config, or comma-separated profile string</TableCell>
    <TableCell>`P240p30fps16x9,P360p30fps16x9,P720p30fps16x9`</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>`-nvidia`</TableCell>
    <TableCell>Comma-separated GPU IDs for NVIDIA hardware transcoding</TableCell>
    <TableCell>Omit for CPU</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>`-concurrentSessions`</TableCell>
    <TableCell>Number of simultaneous transcoding sessions</TableCell>
    <TableCell>`1`</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>`-live`</TableCell>
    <TableCell>Simulate live stream (queue segments one-by-one)</TableCell>
    <TableCell>`true`</TableCell>
  </TableRow>
</StyledTable>

<Note>
  Omit `-nvidia` if you are testing CPU transcoding. With NVIDIA hardware, provide the GPU ID (usually `0` for a single GPU, `0,1` for two GPUs).
</Note>

### Live mode vs VOD mode

By default, `-live=true` queues segments as they would arrive in a live stream — one every 2 seconds. This is the relevant test for network workloads.

To test video-on-demand (all segments queued at once), set `-live=false`:

```bash icon="terminal"
livepeer_bench \
    -in bbb/source.m3u8 \
    -live=false
```

For session limit planning, always use the default live mode.

## Reading the output

The first output is a summary table showing your configuration:

``` icon="terminal"
*---------------------*----------------------------------------------*
| Source File         | .../bbb/source.m3u8                          |
| Transcoding Options | 240p0,360p0,480p0,720p0                      |
| Concurrent Sessions | 4                                            |
| Live Mode           | true                                         |
*---------------------*----------------------------------------------*
```

While transcoding runs, per-segment metrics stream as CSV:

``` icon="terminal"
timestamp,session,segment,seg_dur,transcode_time,frames
2026-03-13 10:00:01.141,0,0,2,0.2545,60
2026-03-13 10:00:03.001,0,1,2,0.1107,60
2026-03-13 10:00:04.998,0,2,2,0.0938,60
...
```

{/* REVIEW: The CSV now includes a `frames` column not present in v1 docs. Verified from livepeer_bench.go source (March 2026). Confirm with Rick that this is stable and whether any downstream tooling expects the old format. */}

When all sessions finish, the summary table is printed:

``` icon="terminal"
*------------------------------*---------------------*
| Concurrent Sessions          | 4                   |
| Total Segs Transcoded        | 120                 |
| Real-Time Segs Transcoded    | 120                 |
| * Real-Time Segs Ratio *     | 1                   |
| Total Source Duration        | 240s                |
| Total Transcoding Duration   | 18.42s              |
| * Real-Time Duration Ratio * | 0.0768              |
*------------------------------*---------------------*
```

### What the ratios mean

**Real-Time Segs Ratio** — the fraction of segments transcoded before their real-time deadline. A ratio of `1` means every segment was done before the next one arrived. A ratio below `1` means some segments fell behind.

**Real-Time Duration Ratio** — total transcoding time divided by total source duration. This is the primary capacity metric:

<StyledTable variant="bordered">
  <TableRow header>
    <TableCell header>Ratio</TableCell>
    <TableCell header>Meaning</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>&lt; 1.0</TableCell>
    <TableCell>Transcoding faster than real time — GPU has headroom</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>= 1.0</TableCell>
    <TableCell>Exactly real time — no headroom, any spike will cause lag</TableCell>
  </TableRow>
  <TableRow>
    <TableCell>&gt; 1.0</TableCell>
    <TableCell>Slower than real time — GPU is overloaded at this session count</TableCell>
  </TableRow>
</StyledTable>

**Production threshold:** A ratio ≤ 0.8 leaves approximately 20% headroom for upload/download overhead and transient load spikes. The last concurrent session count at which you achieve ≤ 0.8 is your **hardware session limit**.

## Session scaling test

Run the benchmark across increasing concurrent session counts to find your hardware ceiling:

```bash icon="terminal"
#!/bin/bash
for i in {1..20}
do
    livepeer_bench \
        -in bbb/source.m3u8 \
        -transcodingOptions transcodingOptions.json \
        -nvidia 0 \
        -concurrentSessions $i |& grep "Duration Ratio" >> bench.log
done
```

This appends one line per session count to `bench.log`. The output looks like:

``` icon="terminal"
| * Real-Time Duration Ratio * | 0.058  |    # 1 concurrent session
| * Real-Time Duration Ratio * | 0.114  |    # 2 concurrent sessions
| * Real-Time Duration Ratio * | 0.421  |    # 3 concurrent sessions
| * Real-Time Duration Ratio * | 0.783  |    # 4 concurrent sessions
| * Real-Time Duration Ratio * | 1.102  |    # 5 concurrent sessions ← exceeds threshold
...
```

In this example, the hardware session limit is **4** — the last count where ratio ≤ 0.8.

**Adjust the loop range** if your GPU handles more sessions. If the ratio is still below 1.0 at 20 sessions, increase the range — replace `{1..20}` with `{1..40}` or higher.

### Multiple GPUs

For identical GPUs: benchmark with a single GPU, then multiply by the number of GPUs.

For mixed GPUs (e.g. an RTX 4090 and an A4000 in the same machine): benchmark each GPU separately by passing its individual ID to `-nvidia`. Then sum the limits.

```bash icon="terminal"
# Benchmark GPU 0 only
livepeer_bench -in bbb/source.m3u8 -transcodingOptions transcodingOptions.json -nvidia 0 -concurrentSessions 4

# Benchmark GPU 1 only
livepeer_bench -in bbb/source.m3u8 -transcodingOptions transcodingOptions.json -nvidia 1 -concurrentSessions 4
```

### CPU-only transcoding

Omit the `-nvidia` flag. Start at `-concurrentSessions 1` and increase. CPU transcoding is significantly slower per session — the ratio will exceed 1.0 at much lower concurrent counts.

{/* REVIEW: v1 docs suggest CPU transcoding supports ~5 sessions as a rule of thumb. This figure dates from 2021 hardware. Modern CPUs (Ryzen 9, Threadripper) may handle more. Community to confirm current rule of thumb. */}

## Exporting results for analysis

Pipe the full benchmark output to a CSV file for further analysis:

```bash icon="terminal"
livepeer_bench \
    -in bbb/source.m3u8 \
    -transcodingOptions transcodingOptions.json \
    -nvidia 0 \
    -concurrentSessions 4 > output.csv
```

The CSV includes per-segment rows plus the summary table at the end. Strip the summary table rows if processing only segment data.
