Hardware is the primary constraint for every workload type on Livepeer. This page is a practical
reference: GPU support policy, session limits by card tier, VRAM requirements by AI pipeline, the
minimum system stack, and the checklist to run before activating on the network.

For help choosing the right path before setup, see <LinkArrow href="/v2/orchestrators/guides/deployment-details/setup-options" label="Setup Options" newline={false} />. For measuring actual GPU transcoding capacity, see <LinkArrow href="/v2/orchestrators/guides/deployment-details/benchmarking" label="Benchmarking" newline={false} />.

<CustomDivider middleText="GPU Support" style={{margin: "-1rem 0 -2rem 0"}} />

## GPU Vendor Support

NVIDIA is the supported GPU vendor for go-livepeer. AMD and Intel GPUs are not supported for
hardware-accelerated transcoding or AI inference. NVIDIA CUDA is required for both NVENC video
transcoding and all AI inference pipelines.

### Driver requirements

Verify NVIDIA drivers are installed and current before anything else:

```bash icon="terminal"
nvidia-smi
# Output shows: driver version, CUDA version, GPU name, memory
# e.g. NVIDIA-SMI 550.54.14    Driver Version: 550.54.14    CUDA Version: 12.4
```

If `nvidia-smi` fails or is missing, install NVIDIA drivers for the target OS before proceeding.
go-livepeer cannot use the GPU without working NVIDIA drivers.

**Minimum CUDA version:** 12.0. Earlier versions are not supported by the AI runner containers.

## NVENC/NVDEC Session Limits

NVIDIA consumer GPUs enforce **concurrent encoding session limits** via NVENC (hardware video
encoder) and NVDEC (hardware video decoder). These limits directly constrain how many simultaneous
transcoding streams a card can handle in hardware mode.

<Warning>
Consumer NVIDIA cards (GeForce RTX series) are capped at a small number of concurrent NVENC sessions
by default. Data centre cards (A-series, L-series, H-series) have no such cap.
</Warning>

Consumer cards are typically limited to **2-3 concurrent NVENC encode sessions** before throttling
or failing with `NvEncOpenEncodeSessionEx` errors. This is a hardware-enforced licensing restriction,
not a software limitation in go-livepeer.

The official NVIDIA matrix shows NVENC and NVDEC session counts per GPU generation:

**[developer.nvidia.com/video-encode-decode-gpu-support-matrix](https://developer.nvidia.com/video-encode-decode-gpu-support-matrix)**

## Hardware Tiers by Workload

### Video transcoding

Video transcoding is NVENC/NVDEC throughput and memory bandwidth. Large VRAM is not required -
model loading is not involved.

**CPU transcoding:** Possible without a GPU, but throughput is significantly lower. Practical
capacity is low single-digit concurrent sessions, varies heavily by CPU generation and codec mix.
Not competitive at scale.

### Real-time AI inference (Cascade)

Real-time video AI has stricter requirements than batch AI:

- **Minimum VRAM:** 12-16 GB (StreamDiffusion with SD 1.5)
- **Competitive VRAM:** 24 GB (StreamDiffusion with SDXL)
- **Additional requirements:** fast PCIe bandwidth for frame buffer throughput; low-latency memory
- **Frame buffer overhead:** add 1-2 GB on top of the model's base VRAM footprint

Real-time pipelines require models warm at all times. A cold start during an active stream causes
visible interruption for end users. See <LinkArrow href="/v2/orchestrators/guides/workloads-and-ai/realtime-ai-setup" label="Realtime AI Setup" newline={false} /> for full runtime requirements.

**Storage note:** AI model weights are large. A single SDXL model is approximately 7-8 GB.
Running multiple pipelines requires fast storage (NVMe) to load models without introducing latency
spikes. Budget at least 100 GB of fast storage per 4-5 models served simultaneously.

**Network note:** Latency to Gateways matters more than raw throughput for AI jobs. A 100 Mbps
connection with sub-20ms latency to major regions outperforms a 1 Gbps connection with high
latency for real-time AI workloads.

<CustomDivider middleText="Pre-launch Checklist" style={{margin: "-1rem 0 -2rem 0"}} />

## Checklist Before Going Live

Run through this before activating on the network:

<StyledSteps>

  <StyledStep title="Verify GPU is visible">

    ```bash icon="terminal"
    nvidia-smi
    # Should output: GPU name, driver version, CUDA version, memory used/total
    # If this fails: drivers are not installed correctly
    ```

  </StyledStep>

  <StyledStep title="Verify Docker can access the GPU">

    ```bash icon="terminal"
    docker run --rm --gpus all nvidia/cuda:12.0-base nvidia-smi
    # Should output the same as nvidia-smi above
    # If this fails: NVIDIA Container Toolkit is not installed or configured
    ```

  </StyledStep>

  <StyledStep title="Install NVIDIA Container Toolkit if needed">

    ```bash icon="terminal"
    # Ubuntu 22.04 / 24.04
    curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
      sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
    curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
      sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
      sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
    sudo apt-get update && sudo apt-get install -y nvidia-container-toolkit
    sudo nvidia-ctk runtime configure --runtime=docker
    sudo systemctl restart docker
    ```

    See the [NVIDIA Container Toolkit install guide](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/install-guide.html) for other distributions.

  </StyledStep>

  <StyledStep title="Confirm required ports are open">

    ```bash icon="terminal"
    # Default go-livepeer Orchestrator ports:
    # 8935 - Orchestrator RPC (must be reachable from the internet)
    # 7935 - CLI HTTP interface (localhost only; not exposed externally)
    sudo ufw status
    # or: sudo iptables -L
    ```

    Port 8935 must be reachable from the internet. If behind NAT, configure port forwarding.

  </StyledStep>

  <StyledStep title="Confirm Arbitrum RPC access">

    An Arbitrum One RPC endpoint is required. Options:

    - Public: `https://arb1.arbitrum.io/rpc`
    - Alchemy, Infura, QuickNode (recommended for production - more reliable)

    ```bash icon="terminal"
    curl -s https://arb1.arbitrum.io/rpc \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_blockNumber","params":[],"id":1}' | python3 -m json.tool
    ```

  </StyledStep>

  <StyledStep title="Check static IP or DDNS">

    The `serviceAddr` (the address Gateways use to reach the Orchestrator) must be stable.
    Home connections with dynamic IPs need DDNS or a static IP via the ISP. Changing IP without
    updating `serviceAddr` causes Gateways to lose contact.

  </StyledStep>

  <StyledStep title="Confirm ETH balance on Arbitrum">

    The Orchestrator wallet needs ETH on Arbitrum One before activation. Budget at least 0.01 ETH
    for the activation transaction and initial reward calls. See <LinkArrow href="/v2/orchestrators/guides/staking-and-rewards/payments" label="Payments" newline={false} /> for ongoing ETH requirements.

  </StyledStep>

</StyledSteps>
