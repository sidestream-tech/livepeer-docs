The session limit is the ceiling on how many concurrent transcoding streams a node will accept.
Too low and capacity goes unused. Too high and the node falls behind - segments pile up, Gateways
receive errors, and the performance reputation suffers.

This page explains how to calculate both bandwidth and hardware limits, then combine them into the
right `-maxSessions` value.

<Note>
Complete <LinkArrow href="/v2/orchestrators/guides/deployment-details/benchmarking" label="Benchmarking" newline={false} /> first. That process produces the hardware limit number this page uses as an input.
</Note>

<CustomDivider middleText="Why It Matters" style={{margin: "-1rem 0 -2rem 0"}} />

## Session Limit Behaviour

go-livepeer defaults to a maximum of **10 concurrent sessions** (`MaxSessions = 10`). When an
incoming stream request would exceed this limit, the Orchestrator returns an `OrchestratorCapped`
error and the Gateway stops routing work until capacity frees up.

{/* REVIEW: `OrchestratorCapped` error string is from v1 docs. Verify this is still the correct error in current go-livepeer when MaxSessions is exceeded. Rick to check against current source. */}

Setting `-maxSessions` based on actual hardware and bandwidth provides two benefits:

- **Protects quality** - prevents the node from accepting more work than it can process in real
  time, avoiding failed transcode jobs and reputational score damage with Gateways
- **Maximises utilisation** - a properly calibrated limit uses as much hardware capacity as the
  node can reliably deliver

renditions). The bandwidth requirement per stream depends on the rendition profile set being used.

**Upload bandwidth per stream:** approximately **5.7 Mbps** (sum of all output rendition bitrates).

**Download bandwidth per stream:** depends on source stream resolution. For a 720p source at 3 Mbps,
download is approximately 3 Mbps per stream. For a 1080p source, download could be 6 Mbps or more.

For planning purposes, budget **~6 Mbps symmetric** per concurrent stream to cover both source
download and output upload with margin.

### Estimating stream count from bandwidth

```text icon="terminal"
Bandwidth limit = available bandwidth (Mbps) ÷ bandwidth per stream (Mbps)
```

The 20% burst margin accounts for segments arriving in bursts rather than at perfectly uniform
intervals. Most operators use the conservative figure when setting `-maxSessions`.

<Warning>
These are theoretical limits. ISP connections may be asymmetric (higher download than upload). Use
the lower of upload and download rates for the bandwidth limit calculation. Many residential
connections with 100 Mbps download have only 20-30 Mbps upload - the upload constraint dominates
in those cases.
</Warning>

## Calculating Session Limit

The session limit is the **minimum** of the hardware limit and the bandwidth limit:

```text icon="terminal"
session limit = min(hardware limit, bandwidth limit)
```

**Hardware limit** - from <LinkArrow href="/v2/orchestrators/guides/deployment-details/benchmarking" label="Benchmarking" newline={false} />: the last concurrent session count where Real-Time Duration Ratio was ≤ 0.8.

**Bandwidth limit** - calculated above from the available symmetric bandwidth.

**Example calculation:**

## Setting the Session Limit

Pass `-maxSessions` to the `livepeer` binary. It applies to both Orchestrators and Transcoders.

### Combined Orchestrator and Transcoder

```bash icon="terminal"
livepeer \
    -orchestrator \
    -transcoder \
    -maxSessions 12 \
    -network arbitrum-one-mainnet \
    # ... other flags
```

### Split setup (separate Orchestrator and Transcoder)

Set `-maxSessions` on both nodes. The Orchestrator uses it to know total capacity; the Transcoder
uses it to control how many concurrent jobs it accepts.

```bash icon="terminal"
# Orchestrator node
livepeer \
    -orchestrator \
    -maxSessions 12 \
    -network arbitrum-one-mainnet

# Transcoder node (separate machine or process)
livepeer \
    -transcoder \
    -maxSessions 12 \
    -orchAddr <orchestrator-ip>:8935
```

## Tuning After Going Live

The benchmark gives an estimate. Real network conditions introduce variables not captured in the
benchmark:

- Actual segment sizes and bitrates from Gateways vary from the test stream
- Upload latency and jitter add overhead beyond raw bandwidth
- `Reward()` calls and ticket redemptions consume CPU and network intermittently

Start with the benchmark-derived limit and monitor. Key metrics to watch after activation:

  <thead>
    <TableRow header>
      <TableCell header>Metric</TableCell>
      <TableCell header>What to watch</TableCell>
    </TableRow>
  </thead>
  <tbody>
    <TableRow>
      <TableCell>`livepeer_transcode_duration_seconds`</TableCell>
      <TableCell>Should stay below segment duration (2 seconds) per session</TableCell>

    <TableRow>

      <TableCell>Drops below 1 when segments are falling behind real time</TableCell>
    </TableRow>
    <TableRow>
      <TableCell>Gateway job rejection rate</TableCell>

    </TableRow>
    <TableRow>
      <TableCell>CPU/GPU utilisation</TableCell>
      <TableCell>Sustained >95% utilisation signals the limit should be reduced</TableCell>
    </TableRow>
  </tbody>
</StyledTable>

If the node consistently handles work without degradation, `-maxSessions` can be incremented.
Adjust based on observed performance, not theoretical headroom.

## AI Inference and Session Limits

`-maxSessions` applies to video transcoding sessions only. AI inference capacity is configured
separately through the `capacity` field in `aiModels.json` per pipeline entry:

```json icon="code" title="aiModels.json"
{
    "pipeline": "text-to-image",
    "model_id": "SG161222/RealVisXL_V4.0_Lightning",
    "price_per_unit": 4768371,
    "warm": true,
    "capacity": 2
}
```

The `capacity` field limits concurrent inference requests to that pipeline. AI capacity is
bounded by VRAM rather than by stream count or bandwidth. For the full AI capacity reference,
see <LinkArrow href="/v2/orchestrators/guides/workloads-and-ai/model-vram-reference" label="Models and VRAM Reference" newline={false} />.
