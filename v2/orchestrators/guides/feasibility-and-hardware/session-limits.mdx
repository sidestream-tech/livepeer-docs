Your session limit is the ceiling on how many concurrent transcoding streams your node will accept. Set it too low and you leave capacity unused. Set it too high and your node falls behind — segments pile up, gateways get errors, and your performance reputation suffers.

This page explains how to calculate both your bandwidth limit and hardware limit, then combine them into the right `-maxSessions` value.

<Note>
  Complete [Benchmarking Your Setup](/v2/orchestrators/guides/feasibility/benchmarking) first. That process gives you the hardware limit number this page uses as an input.
</Note>

---

## Why session limits matter

Go-livepeer has a default maximum of **10 concurrent sessions** (`MaxSessions = 10`). When an incoming stream request would exceed this limit, the orchestrator returns an `OrchestratorCapped` error and the gateway stops routing work to you until capacity frees up.

{/* REVIEW: `OrchestratorCapped` is the error name from v1 docs. Verify this string is still the correct error returned in current go-livepeer when MaxSessions is exceeded. Rick to check against current source — couldn't confirm in source scrape due to GitHub API limitations. */}

Setting a limit based on your actual hardware and bandwidth has two benefits:

- **Protects quality** — prevents your node from accepting more work than it can process in real time, which would cause failed transcode jobs and hurt your reputation score with gateways.
- **Maximises utilisation** — a properly calibrated limit means you use as much of your hardware as you can reliably deliver.

## Bandwidth per stream

Every transcoding job requires both download (receiving the source stream) and upload (delivering output renditions). The bandwidth requirement per stream depends on the rendition profile set being used.

The current standard output rendition set in `transcodingOptions.json` is:

| Profile | Resolution | Output bitrate |
|---|---|---|
| `240p0` | 426×240 | 250 kbps |
| `360p0` | 640×360 | 800 kbps |
| `480p0` | 854×480 | 1,600 kbps |
| `720p0` | 1,280×720 | 3,000 kbps |
| **Total output** | | **5,650 kbps** |

**Upload bandwidth per stream:** approximately **5.7 Mbps** (sum of all output rendition bitrates).

**Download bandwidth per stream:** depends on source stream resolution. For a 720p source at 3,000 kbps, download is approximately **3 Mbps** per stream. For a 1080p source, download could be **6 Mbps** per stream or more.

For planning purposes, budget **~6 Mbps symmetric** per concurrent stream to cover both source download and output upload with margin.

### Estimating your stream count from bandwidth

```
Bandwidth limit = available bandwidth (Mbps) ÷ bandwidth per stream (Mbps)
```

**Examples:**

| Available bandwidth | Bandwidth limit (streams) | With ~20% burst margin |
|---|---|---|
| 100 Mbps symmetric | ~16 streams | ~13 streams |
| 500 Mbps symmetric | ~83 streams | ~66 streams |
| 1 Gbps symmetric | ~166 streams | ~133 streams |

The 20% burst margin accounts for segments arriving in bursts rather than perfectly spaced. Most operators use the conservative figure when setting `-maxSessions`.

<Warning>
  These are theoretical bandwidth limits. Your actual ISP connection may have asymmetric speeds (higher download than upload). Use the lower of upload and download rates for calculating your bandwidth limit. Many residential connections with 100 Mbps download have only 20–30 Mbps upload — the upload constraint dominates.
</Warning>

## Deriving your session limit

Your session limit is the **minimum** of your hardware limit and your bandwidth limit:

```
session limit = min(hardware limit, bandwidth limit)
```

**Hardware limit** — the last concurrent session count in your benchmark where the Real-Time Duration Ratio was ≤ 0.8. See [Benchmarking Your Setup](/v2/orchestrators/guides/feasibility/benchmarking).

**Bandwidth limit** — calculated above from your available symmetric bandwidth.

**Example:**

| Input | Value |
|---|---|
| Hardware limit (from benchmarking) | 12 sessions |
| Available bandwidth | 100 Mbps symmetric |
| Bandwidth limit (100 Mbps ÷ 6 Mbps, with 20% margin) | ~13 sessions |
| **Session limit** | **min(12, 13) = 12** |

In this example, hardware is the binding constraint. Use `-maxSessions 12`.

## Setting the session limit

Pass `-maxSessions` to the `livepeer` binary. It applies to both orchestrators and transcoders.

### Combined orchestrator + transcoder

```bash
livepeer \
    -orchestrator \
    -transcoder \
    -maxSessions 12 \
    -network arbitrum-one-mainnet \
    # ... other flags
```

### Split setup (orchestrator + separate transcoder)

Set `-maxSessions` on both the orchestrator and the transcoder nodes. The orchestrator uses it to know its total capacity; the transcoder uses it to control how many concurrent jobs it accepts.

```bash
# Orchestrator node
livepeer \
    -orchestrator \
    -maxSessions 12 \
    -network arbitrum-one-mainnet

# Transcoder node (separate machine or process)
livepeer \
    -transcoder \
    -maxSessions 12 \
    -orchAddr <orchestrator-ip>:8935
```

## Tuning after going live

The benchmark gives you a starting estimate. Real network conditions introduce variables not captured in the benchmark:

- Actual segment sizes and bitrates from gateways vary
- Upload latency and jitter add overhead beyond raw bandwidth
- Reward() calls and ticket redemptions consume CPU/network intermittently

**Start with your benchmark-derived limit and watch your metrics.** Key things to monitor after going live:

| Metric | What to watch |
|---|---|
| `livepeer_transcode_duration_seconds` | Should stay below segment duration (2s) per session |
| `livepeer_real_time_seg_transcoded` | Drops below 1 when segments are falling behind |
| Gateway job rejection rate | If gateways stop sending you work, check OrchestratorCapped errors in logs |
| CPU/GPU utilisation | Sustained >95% utilisation is a signal to reduce sessions |

If your node is consistently handling work without degradation, you can experiment with incrementally raising `-maxSessions`. Adjust based on observed performance, not just theoretical headroom.

## AI workloads and session limits

`-maxSessions` applies to video transcoding sessions. AI inference capacity is managed differently — through the `capacity` field in `aiModels.json` per pipeline entry:

```json
{
    "pipeline": "text-to-image",
    "model_id": "SG161222/RealVisXL_V4.0_Lightning",
    "price_per_unit": 4768371,
    "warm": true,
    "capacity": 2
}
```

The `capacity` field limits concurrent inference requests to that pipeline. AI capacity is bounded by VRAM rather than by stream count or bandwidth. For the full AI capacity reference, see [Models and VRAM Reference](/v2/orchestrators/guides/workloads/model-vram-reference).
