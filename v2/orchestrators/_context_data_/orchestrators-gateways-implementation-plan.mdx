### 1.1 Fix the NVIDIA Driver Factual Error

**File:** `v2/gateways/run-a-gateway/install/docker-install.mdx`  
**Also check:** Any install prerequisite pages under `v2/gateways/run-a-gateway/install/`

**What to do:**  
Search the entire gateways section for any mention of NVIDIA drivers as a gateway prerequisite. 
Gateways do not require NVIDIA drivers. Remove or correct every instance.

The correct rule is:
- **Orchestrators / GPU Nodes** → NVIDIA drivers required
- **Gateways** → No NVIDIA drivers needed

Add a callout in the gateway install prerequisites to make this explicit and prevent future confusion:

```mdx
<Note>
  Gateways do not require NVIDIA GPU drivers. GPU requirements apply to 
  Orchestrators only. See the [Orchestrator setup guide](/v2/orchestrators) 
  if you are running GPU compute.
</Note>
```

**Why this matters:** This came directly from the contributor who reviewed the section. Factual errors in install prerequisites destroy trust immediately and silently cause failed setups.

#### Step 1 - Create and fund your ETH account

This is the actual first entrypoint. The ETH account creation is currently missing or buried. 
It needs to be Step 1, not a reference page.

```mdx
## Step 1: Create your Ethereum account

You need an Arbitrum One ETH account to operate a node on-chain. If you already have one, 
skip to [funding it](#fund-your-account).

### Create a new keystore

```bash
# Using go-livepeer to generate a keystore
./livepeer -network arbitrum-one-mainnet -ethKeystorePath ~/.lpData/keystore
````

Follow the prompts to set a password. Store your keystore and password securely - 
this controls your node's funds and identity.

<Warning>
  Never commit your keystore or password to version control. 
  Add `~/.lpData/keystore` to your `.gitignore`.
</Warning>

### Fund your account

Your node needs ETH on Arbitrum One for gas and ticket payments.

- Minimum recommended: **0.1 ETH** to get started
- Use the [Arbitrum Bridge](https://bridge.arbitrum.io/) to move ETH from mainnet
- See [Arbitrum exchanges](/v2/gateways/references/arbitrum-exchanges) for other options
```

#### Step 3 - Pull and configure

Slot eliteproxy's Docker compose here. Add annotation comments to explain each environment variable. 
Do not leave compose variables unexplained - this is where first-time operators will get lost.

Example annotation pattern:

```mdx
## Step 3: Configure and run

```yaml
# docker-compose.yml
services:
  orchestrator:
    image: livepeer/go-livepeer:latest
    environment:
      # Your funded Arbitrum ETH address
      - ETH_KEYSTORE_PATH=/keystore
      - ETH_PASSWORD_PATH=/password
      # Pricing: cost per pixel in wei. 1000 = ~$0.000001 per pixel
      - PRICE_PER_UNIT=1000
      # Max concurrent transcoding sessions
      - MAX_SESSIONS=10
      # Your publicly accessible service URI (must be reachable by gateways)
      - SERVICE_URI=https://your-orchestrator.example.com:8935
    volumes:
      - ~/.lpData/keystore:/keystore:ro
      - ~/.lpData/password:/password:ro
```
```

## Step 4: Register as an orchestrator

### 3.3 Add the Job Type Concept Page

**File to create:** `v2/orchestrators/about-orchestrators/job-types.mdx`

This is the explanation page that feeds the three-path quickstart. It answers: why do the three 
configuration paths exist? What's different between them?

```mdx
---
title: Types of Orchestrator Jobs
description: Understanding the difference between Transcoding, Realtime AI, and Batch AI job configurations.
sidebarTitle: Job Types
---

Livepeer Orchestrators can serve three distinct types of compute jobs, each requiring 
different configuration and (in some cases) different software.

## Transcoding

Video transcoding converts streams between formats, bitrates, and resolutions. 
This is Livepeer's original use case.

- **GPU:** Optional (CPU transcoding available, GPU strongly recommended for throughput)
- **Additional software:** None beyond go-livepeer
- **Typical use:** Live video platforms, VOD pipelines

## Realtime AI

Real-time video-to-video AI pipelines process live video streams through AI models 
(e.g., depth estimation, style transfer, segmentation) with sub-second latency requirements.

- **GPU:** Required
- **Additional software:** `mediamtx` for stream handling; see [box.md](https://github.com/livepeer/go-livepeer/blob/master/box/box.md)
- **Typical use:** Live AI video effects, interactive video

<Note>
  Realtime AI setup is more involved than transcoding. Review the 
  [box documentation](https://github.com/livepeer/go-livepeer/blob/master/box/box.md) 
  before proceeding.
</Note>

## Batch AI

Asynchronous AI inference over video or image content. Jobs are queued and processed 
without latency constraints.

- **GPU:** Required
- **Additional software:** Pipeline-dependent
- **Typical use:** Post-production AI, content analysis, large-scale inference

## Structural Note: Studio vs go-livepeer Separation

eliteproxy's most important systemic observation:

> *"IMO the Livepeer Studio guides should be their own community of practice, for Studio's own API. 
> They do not communicate network value to technically minded builders."*

This isn't a Phase 1 change - it's a longer-term IA decision - but the groundwork should be laid now.

**Immediate action:** Ensure every guide that is Studio-specific is clearly labelled as such. 
Add a callout at the top of Studio-specific pages:

```mdx
<Info>
  This guide uses the **Livepeer Studio API**, a managed service built on the Livepeer network. 
  If you want to interact with the network directly using go-livepeer, 
  see the [Gateway guide](/v2/gateways) or [Orchestrator guide](/v2/orchestrators).
</Info>
```

This makes the distinction visible without restructuring the entire nav yet.
