# Personas and Journeys

### PART 1 — PERSONA RESEARCH
Sources: Livepeer Forum, Livepeer.org/orchestrate, Mirror.xyz, Messari Q1–Q3 2025, Live Pioneers KYO series, Udemy course descriptions, Cloud SPE blog, go-livepeer GitHub, community Discord intelligence, RFP documentation.

Persona A — The Solo GPU Operator ("The Miner")
Background
Former crypto miner (ETH, GPU mining) who transitioned to Livepeer after Ethereum's proof-of-stake switch made GPU mining unprofitable. Has 1–4 consumer NVIDIA GPUs (RTX 2060–3080 class), typically running on home broadband or a small VPS. Technically capable but not a devops professional. Usually solo, sometimes part of Video Miner or Titan Node pool.
Motivations

Passive income from existing hardware (ETH fees + LPT staking rewards)
Crypto community participation — this is part identity, part finance
"Set and forget" operation; doesn't want daily hands-on management

Entry Points

GPU mining communities (reddit r/gpumining, Discord, Telegram)
Udemy course ("How to start a Livepeer Orchestrator and make passive income")
Video Miner and Titan Node websites
YouTube "passive income with GPU" content
Live Pioneers community content

Key Questions at Each Stage

Orienting: What is this and is it worth my time? How much will I earn?
Decision: Should I run a solo orchestrator or join a pool? What LPT do I need?
Activation: How do I install go-livepeer? How do I configure it? What flags do I need?
Operational: Why aren't I getting jobs? How do I set price per pixel? How do I monitor earnings?
Deepening: Should I add AI pipelines to increase income? How?
Reference: CLI flags, Livepeer Explorer, pricing tools

Pain Points (confirmed from forum + community sources)

LPT acquisition barrier — must acquire/stake LPT before earning; this blocks many entrants
Pool vs solo confusion — unclear in existing docs when you need full orchestrator vs pool worker
Price per pixel — no guidance on competitive pricing; operators discuss this extensively on Discord
Port forwarding / networking — serviceAddr setup trips up home users
"Orchestrator not available" errors on startup — recurrent newbie forum complaint
Monitoring — Grafana setup is complex; most users want a simpler status check

Hardware Profile
RTX 2060–3080 (8–12GB VRAM), 1–4 GPUs, consumer home machine or budget VPS

Persona B — The Pool Worker ("The Easy Earner")
Background
Has one or more consumer GPUs and wants to earn from Livepeer without the complexity of running a full orchestrator node (which requires significant LPT stake to compete). Joins Video Miner, Titan Node, LivePool, or another community pool as a worker node. Contributes GPU compute; the pool orchestrator handles on-chain operations.
Motivations

Earn passive income from GPU without needing to acquire large LPT stake
Simplicity — just run a transcoder/worker process

Entry Points

Video Miner website, Titan Node marketing, community Discord
"Know Your Orchestrator" content from Live Pioneers
Pool-specific documentation (outside livepeer.org)

Key Questions

Orienting: What's the difference between a pool worker and a solo orchestrator?
Activation: How do I join a pool? Do I run go-livepeer differently?
Operational: How do I check my earnings? How does the pool distribute rewards?

Pain Points

The distinction between "running an orchestrator" and "joining a pool as a worker" is buried in docs
Community pools page lists pools but doesn't explain the operational model
No single clear "join a pool" guide — most pool documentation lives outside Livepeer's docs

Hardware Profile
Single GPU (RTX 1070Ti–3070, 8GB VRAM), often home machine

Persona C — The Infrastructure Engineer ("The Pro Operator")
Background
Already running a Livepeer orchestrator with 4–20 GPUs, often in a data centre or hosted environment. Strong devops background — comfortable with Docker, Kubernetes (or k3s), Prometheus/Grafana monitoring, Linux system administration. Looking to optimise earnings and expand into AI workloads.
Motivations

Maximise ROI from GPU infrastructure
Add AI inference pipelines on top of existing transcoding for additional fee revenue
Build reliable, monitored, scalable infrastructure

Entry Points

Already deep in Livepeer ecosystem — uses explorer.livepeer.org, Livepeer Discord orchestrating channel
Follows Livepeer blog, Messari reports
Cloud SPE blog posts (e.g., Ollama-based LLM runner guide)

Key Questions

Operational: How do I set up AI Runner containers? What VRAM do I need per pipeline?
Deepening: How do I set up remote workers (BYOC)? How do I host and manage models?
Deepening: How do I run an orchestrator pool and let others join?
Reference: CLI flags, aiModels.json schema, pricing reference for AI jobs vs transcoding

Pain Points

AI pipeline configuration (aiModels.json) is underdocumented
Remote worker setup requires Discord spelunking
No clear earnings optimisation guide for operators with mixed workloads
Run-a-pool documentation is thin
BYOC/model hosting guidance scattered across multiple sources

Hardware Profile
4–20+ NVIDIA GPUs (RTX 3090, A5000, or server-class), data centre or hosted provider, Docker-native setup

Persona D — The Enterprise / Data Centre Operator ("The Business")
Background
Runs professional GPU infrastructure — either a GPU cloud provider evaluating Livepeer as an additional workload, or a business building Livepeer-native compute services. Evaluating at a business level: ROI, compliance, SLA reliability, programmatic management.
Motivations

New revenue stream on existing GPU assets
Build a Livepeer compute business (e.g., Livepeer.Cloud SPE model)
Evaluate protocol economics before committing significant capital

Entry Points

Livepeer Foundation BD outreach, conferences
Messari State of Livepeer reports
livepeer.org/orchestrate
Seeing Livepeer mentioned in AI compute context (Cascade, Daydream)

Key Questions

Orienting: What is the business model? What are the economics at scale? How does LPT stake affect earnings?
Decision: Is this worth running as a business? What's the regulatory/compliance posture?
Activation: How do we provision infrastructure at scale? What's the data centre setup guide?
Operational: How do we manage multi-region orchestrators? How do we publish offerings?
Deepening: How do we run a pool and onboard other operators?

Pain Points

No enterprise-facing section in current docs
Economics at scale are not documented (data-centre-setup is a stub)
No programmatic management documentation
Protocol economics scattered across Livepeer.org, forum, Messari — not in docs

Hardware Profile
A100/H100/4090 server GPUs, multi-region, hundreds of GPUs, cloud or colo data centre

Persona E — The AI Compute Specialist ("The AI Native")
Background
ML engineer, AI researcher, or AI startup operator. Came to Livepeer through the AI subnet launch (Q3 2024), Daydream, or Cascade announcements. Interested in Livepeer as a decentralised compute layer for AI inference rather than transcoding. May have no prior Livepeer or crypto background.
Motivations

Earn revenue serving open-source AI models at scale
Access decentralised GPU compute for AI workloads
Build on or contribute to open AI infrastructure (anti-centralisation values)

Entry Points

Livepeer blog posts (Mirror.xyz — "Introducing Livepeer AI")
Real-Time Video AI Bootcamp (Encode Club, Q1 2025)
ComfyStream integration announcements
Daydream / Cascade product announcements
Cloud SPE blog (Ollama runner guide, Nov 2025)

Key Questions

Orienting: How does the Livepeer AI network work? What pipelines are supported? How is pricing set?
Decision: What hardware do I need? What's the VRAM minimum per pipeline?
Activation: Realtime AI quickstart vs batch AI quickstart — which do I choose?
Operational: How do I configure aiModels.json? How do I keep models warm? How do I set pricing?
Deepening: How do I host custom models (BYOC)? How do remote workers scale capacity?
Reference: Pipeline docs, model compatibility list, CLI flags for AI-specific flags

Pain Points

"Realtime AI" vs "Batch AI" distinction not clearly explained before quickstart
aiModels.json format and valid model IDs are not well-documented
Docker container management for AI Runner is complex
Minimum VRAM requirements vary by pipeline/model and change frequently
No pricing guidance for AI jobs (what's competitive? what earns the most?)
"Cascade" pipeline (for streaming AI agents) lacks dedicated setup documentation

Hardware Profile
RTX 4090 (24GB VRAM), A100/H100, cloud instances — high VRAM priority
