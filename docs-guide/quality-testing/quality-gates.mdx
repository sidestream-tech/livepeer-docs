## Gate Layers

### 1) Pre-commit (local)

Entry:

- `.githooks/pre-commit`

Key checks:

- repository structure rules
- style guide checks
- syntax/verification checks (`.githooks/verify.sh`)
- staged test suite (`tests/run-all.js --staged --skip-browser`)
- staged link/domain/WCAG audits
- staged WCAG audit runs `tests/integration/v2-wcag-audit.js --staged --fix --stage --max-pages 10 --fail-impact serious ...`
- script docs enforcement
- pages index synchronization

Blocking policy:

- blocking for structural/style/test failures
- blocking for staged WCAG runtime failures and remaining `serious`/`critical` accessibility findings after conservative autofix
- `color-contrast` findings are advisory-only (non-blocking) in the WCAG audit until color updates are approved
- docs-guide SoT check currently advisory

### 2) PR Changed-File Checks

Entry:

- `tests/run-pr-checks.js`
- `.github/workflows/test-suite.yml` (PR mode)

Key checks:

- style, MDX, spelling, quality, links/imports (changed scope)
- MDX guardrails + docs navigation checks
- script docs checks on changed scripts
- strict link audit on changed docs pages
- docs-guide SoT check currently advisory

Blocking policy:

- blocking for failed required checks
- advisory behavior for integration PR exceptions and docs-guide SoT check

### 3) Browser Validation / Sweeps

Entries:

- `tests/integration/browser.test.js`
- `tools/scripts/test-v2-pages.js`
- `.github/workflows/test-v2-pages.yml`

Blocking policy:

- browser failures are treated as blocking in primary CI flows

### 4) OpenAPI Reference Integrity (V2 + Locales)

Entries:

- `tests/integration/openapi-reference-audit.js`
- `.github/workflows/openapi-reference-validation.yml`

What it validates:

- endpoint-form OpenAPI references in frontmatter and `<OpenAPI ...>` components
- canonical file-path to spec mapping
- exact method+path presence in mapped local spec
- `intra-file-openapi-mismatch` between frontmatter and component in the same page

Run triggers:

- pull requests to `main` and `docs-v2`
- pushes to `main` and `docs-v2`
- daily schedule at `04:35 UTC`
- manual `workflow_dispatch`

Autofix boundaries:

- only method casing normalization
- only single-space `METHOD /path` normalization
- only missing leading slash normalization
- no semantic endpoint-path rewrites

Rolling issue behavior:

- single marker issue `<!-- openapi-reference-audit -->`
- open/update on unresolved failures
- add resolution comment + close when run is clean

Blocking policy:

- unresolved findings are blocking for PR/push workflow outcomes

Triage for `endpoint-not-found-in-spec`:

1. verify `resolvedSpec` from report
2. verify endpoint presence in mapped spec
3. update both frontmatter `openapi` and `<OpenAPI path=...>` together
4. retire endpoint pages and locale variants when endpoint is intentionally removed
5. rerun strict audit until clean

## Guardrails for Docs SoT

Guardrail checks are implemented in:

- `tests/unit/docs-guide-sot.test.js`

Generated-index freshness is checked through:

- `tools/scripts/generate-docs-guide-indexes.js --check`

Advisory rollout intent:

1. advisory in pre-commit + CI summaries
2. switch to blocking once docs-guide structure stabilizes and drift rate is low
