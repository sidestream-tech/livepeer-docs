<Tip>
`lpd` is the repository CLI for setup, local development, test orchestration, hook management, and script execution.
Use this page as both your first-run guide and full command/flag reference.
</Tip>

## What is lpd

`lpd` standardises local docs setup and operations for this repo.
It wraps setup, preview, tests, hooks, and managed scripts behind one command surface.
Behaviour source of truth is the repository-root [`lpd`](../lpd) script.

## Start Here (New Contributor Path)

1. Install/bootstrap the repo CLI environment.
2. Verify your environment and hook state.
3. Start local docs preview.
4. Run baseline checks before opening a PR.

## Installation

Standard install/bootstrap command:

```bash
bash lpd setup --yes
```

What `setup` does:

- Installs `tools` dependencies (`cd tools && npm install`)
- Installs `tests` dependencies (`cd tests && npm install`)
- Installs/updates git hooks (`.githooks/install.sh`)
- Optionally installs `lpd` to your PATH (unless `--no-cli`)

Verify installation:

```bash
lpd version
lpd doctor
```

If PATH has not refreshed yet:

```bash
bash lpd version
bash lpd doctor
```

Common variants:

```bash
lpd setup --no-cli
lpd setup --no-tests
lpd setup --with-mintlify
LPD_INSTALL_DIR="$HOME/.local/bin" lpd setup --yes
```

<Note>
If `lpd setup` updates your shell rc file, open a new terminal or run `source ~/.zshrc` (or your active shell rc) before using plain `lpd`.
</Note>

## First 10 Minutes Workflow

### I just need docs running

```bash
bash lpd setup --yes
lpd dev -- --port 3333
```

### I need to run checks before PR

```bash
lpd doctor --strict
lpd test --staged
lpd ci --skip-browser
```

### I need to run a managed script

```bash
lpd scripts list --group tools
lpd tools dev test-add-callouts -- --help
```

## Troubleshooting

### `lpd` not found

Use the repo-local fallback and verify setup:

```bash
bash lpd setup --yes
bash lpd version
```

If needed, reload your shell rc file (`source ~/.zshrc` or equivalent).

### Hooks mismatch

Check and reinstall hooks:

```bash
lpd hooks status
lpd hooks install
lpd hooks verify
```

### `--json` expectations

Global JSON mode is prefix form: `lpd --json <command> ...`.
Some commands also accept local `--json`.
`lpd info --json` is invalid because `info` only accepts `--help|-h`.

### Port collision on `lpd dev`

Pass a port through the launcher:

```bash
lpd dev -- --port 3334
```

### Slow startup or slow route changes on very large `docs.json`

Use scoped dev mode so Mint loads only the version/language/tab/prefix you are actively editing:

```bash
lpd dev --scoped --scope-version v2 --scope-language en --scope-tab Developers
```

Interactive picker:

```bash
lpd dev --scoped --scope-interactive
```

## JSON Output

When JSON mode is active on finish-based commands, output uses this status envelope:

<Accordion title="JSON Output Envelope" icon="code">
  <ResponseField name="ok" type="boolean">
    `true` for success and `false` for failure.
  </ResponseField>

  <ResponseField name="command" type="string">
    Normalised command label (for example: `test`, `scripts list`, `hooks status`).
  </ResponseField>

  <ResponseField name="message" type="string">
    Human-readable summary of the command result.
  </ResponseField>

  <ResponseField name="exit_code" type="number">
    Exit code returned by the command path.
  </ResponseField>

  <ResponseField name="repo_root" type="string">
    Absolute repo root used during command execution.
  </ResponseField>

  Example:

  ```json
  {
    "ok": true,
    "command": "version",
    "message": "lpd 0.2.0",
    "exit_code": 0,
    "repo_root": "/path/to/livepeer-docs-repo"
  }
  ```

  <Note>
  `lpd info` does not emit a structured JSON envelope.
  Use JSON mode with commands that complete via `finish()`.
  </Note>
