{/*
GitHub Gists can work well for:

- **full MDX layouts/templates** — reusable page shells, callout patterns, frontmatter blocks, component examples, landing page sections
- **VS Code snippets** — repo-specific snippet JSON files, MDX authoring snippets, React/component snippets, Mintlify/MDX boilerplate
- **small reusable content primitives** — tabs, cards, accordions, API example blocks, Mermaid templates

Where they work best:

- versioned reusable text/code blocks
- fast sharing across contributors
- lightweight template library
- copy/paste authoring helpers

Where they are weaker:

- not ideal for **discoverability/governance** at scale
- no strong repo-local coupling by default
- harder to enforce as a true source of truth
- worse than a repo for reviewing changes alongside docs

Best practice split:

- use **the repo** for anything that should be canonical and shipped with the docs
- use **gists** for shared drafts, experiments, snippet libraries, and authoring helpers
- use **repo-local `.vscode/*.code-snippets`** or `.code-snippets` files for team snippets tied to that docs repo

For your use case, the strongest setup is usually:

1. **Repo-native MDX templates** in a `/templates` or `/snippets` folder
2. **Repo-specific VS Code snippets** in `.vscode/livepeer-docs.code-snippets`
3. **Optional Gists** for quick-share patterns or external contributor starter kits

Example uses:

- `quickstart-template.mdx`
- `persona-landing-page.mdx`
- `concept-page-template.mdx`
- `how-to-page-template.mdx`
- `value-response-field` snippet
- Mintlify card-grid snippet
- Mermaid architecture diagram snippet

If you want, I can generate:

- a **GitHub Gist-ready MDX template pack**
- a **repo-specific VS Code snippets file**
- or both, tailored to your Livepeer docs structure. */}

**`fm`** — minimal frontmatter (title + description only).

**Individual field snippets** — each inserts one YAML line with a dropdown of canonical values:

| Prefix | Values |
|--------|--------|
| `pageType` | `landing`, `overview`, `tutorial`, `quickstart`, `how_to`, `concept`, `reference`, `faq`, `troubleshooting`, `changelog`, `glossary`, `guide` |
| `audience` | `developer`, `gateway-operator`, `orchestrator`, `delegator`, `platform-builder`, `community`, `internal`, `general`, `everyone` |
| `status` | `draft`, `review`, `published`, `deprecated`, `archived`, `stub`, `wip`, `needs_update`, `frozen` |
| `purpose` | `landing`, `overview`, `walkthrough`, `how_to`, `explanation`, `reference`, `troubleshooting`, `faq`, `changelog`, `glossary` |
| `lastVerified` | Auto-fills today's date |
